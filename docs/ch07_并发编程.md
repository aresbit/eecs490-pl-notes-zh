# 第7章: 并发编程 (Concurrent Programming)

**并发** (concurrency) 是指程序在重叠的时间段内执行多个计算任务的能力。并发并不要求这些任务在物理上同时执行——只要它们的生命周期存在重叠即可。与之相对，**并行** (parallelism) 是指多个计算任务在物理上真正同时执行，通常需要多核 (multi-core) 处理器或分布式系统。区分这两个概念至关重要: "并发是逻辑上的同时性，并行是物理上的同时性。"

## 7.1 并发编程简介

### 7.1.1 历史背景: 从时钟频率到多核

在 2005 年左右，处理器行业遭遇了著名的"频率之墙" (power wall)。在此之前，摩尔定律主要通过提升时钟频率来兑现——从 1980 年代的数 MHz 一路攀升到 2005 年前后的约 4 GHz。然而，随着频率继续提升，功耗和散热问题变得不可持续。处理器功耗与频率的关系近似为:

$$P \propto f \cdot V^2$$

其中 $$f$$ 是时钟频率，$$V$$ 是电压（随频率提高而提高）。当频率逼近 4 GHz 时，散热密度已经接近核反应堆的水平。

此后，行业转向多核架构: 不再追求单个核心运行得更快，而是在同一芯片上放置多个计算核心。这意味着单个顺序程序不再自动受益于新一代硬件，除非它被重新设计来利用并发和并行。这就是所谓的"免费午餐的终结" (The Free Lunch Is Over, Sutter 2005): 程序员不能再指望新硬件自动让程序跑得更快，而必须主动采用并发编程技术。

### 7.1.2 上下文切换

单个 CPU 核心在任意时刻只能执行一条指令流。操作系统通过**上下文切换** (context switching) 实现多任务的并发执行: 以极高频率（通常在毫秒级）在不同线程之间切换，保存和恢复寄存器状态、程序计数器和栈指针。从用户角度看，多个线程仿佛在"同时"运行，尽管它们实际上是在交错执行 (interleaved execution)。

上下文切换不是免费的。它涉及:
1. **保存上下文**: 将当前线程的寄存器（包括通用寄存器、程序计数器 PC、栈指针 SP、条件码寄存器等）保存到线程的内核栈或线程控制块 (TCB) 中
2. **调度决策**: 操作系统调度器选择一个就绪线程（涉及调度算法如 CFS、优先级队列等）
3. **恢复上下文**: 将被选中线程的寄存器状态从 TCB 加载到物理寄存器
4. **地址空间切换**（仅进程切换需要）: 刷新 TLB、切换页表基址寄存器

线程切换（同一进程内的上下文切换）通常需要几微秒到几十微秒。进程切换由于涉及地址空间切换，成本更高，通常需要几十到几百微秒。在 CPU 密集型程序中，频繁的上下文切换可能带来显著开销: 如果有 100 个线程竞争一个 CPU 核心，每个线程只获得 1% 的 CPU 时间，而上下文切换本身可能消耗 5%~10% 的时间。

### 7.1.3 共享可变状态的挑战

并发编程的核心挑战在于**共享可变状态** (shared mutable state)。当多个线程同时读写同一块内存时，执行顺序的微小变化可能导致截然不同的结果。这就是竞态条件的根源。

考虑一个简单模型: 如果程序有 $$n$$ 个线程，每个线程有 $$k$$ 条指令，且单条指令的执行是原子的，那么可能的交错数量是一个天文数字:

$$\frac{(n \cdot k)!}{(k!)^n}$$

对于只有 2 个线程各 10 条指令的程序，可能的交错数约为 $$184756$$ 种。在如此巨大的状态空间中，错误的交错可能极为罕见，但仍然是可能的——这正是并发 bug 难以重现和调试的根本原因。

### 7.1.4 引用透明与纯函数

**引用透明** (referential transparency) 是函数式编程的核心概念: 一个表达式是引用透明的，当且仅当它可以用其值替换而不改变程序的行为。具有引用透明的函数称为**纯函数** (pure function)——给定相同输入，总是产生相同输出，并且没有副作用。

纯函数天然适合并发编程: 由于不修改共享状态，多个线程可以自由地并行调用纯函数而不需要任何同步。更形式化地说，纯函数满足 Church-Rosser 性质: 无论以何种顺序求值，最终结果总是相同的。

这一洞察是 Berkeley View (Asanovic et al., 2006) 中 13 个并行模式的理论基础。Berkeley View 识别出 13 种"计算基元" (computational dwarfs)，代表了科学计算和工程计算中最常见的计算模式。这些模式涵盖有限状态机、组合逻辑、图遍历、结构化网格、非结构化网格、稠密线性代数、稀疏线性代数、谱方法 (FFT)、N 体问题、蒙特卡洛方法、动态规划、回溯与分支定界，以及图形模型。值得注意的是，在 Berkeley View 的 13 种基元中，只有 MapReduce 是不涉及共享可变状态的。其余 12 种模式都需要某种形式的同步或数据管理来协调并行访问。

## 7.2 Python 中的并行

### 7.2.1 多线程与 GIL

Python 标准库提供了 `threading` 模块来创建和管理线程。基本用法如下:

```python
import threading

def worker(name):
    print(f"线程 {name} 正在运行，ID: {threading.current_thread().ident}")

t1 = threading.Thread(target=worker, args=("A",))
t2 = threading.Thread(target=worker, args=("B",))

t1.start()
t2.start()

t1.join()
t2.join()
```

关键 API:
- `threading.Thread(target, args)`: 创建一个新线程，`target` 指定执行的函数，`args` 传递参数元组
- `.start()`: 启动线程执行（调用操作系统的线程创建原语，分配独立的调用栈，然后执行 `target` 函数）
- `.join()`: 调用线程阻塞直到目标线程结束。`join()` 不仅表示"等待"，它还执行线程资源的回收 (reaping)
- `threading.current_thread()`: 获取当前执行线程的 `Thread` 对象

**CPython 的 GIL (Global Interpreter Lock)** 限制了多线程在 CPU 密集型任务上的并行性。GIL 是一个互斥量，保护对 Python 对象的访问，防止多个线程同时执行 Python 字节码。具体机制是: 每个线程在执行 Python 字节码之前必须先获取 GIL，GIL 每执行约 100 条字节码指令或遇到 I/O 操作时被释放。

GIL 的设计初衷是简化 CPython 解释器的实现——有了 GIL，CPython 的内存管理（特别是引用计数）不需要支持细粒度的并发，因为所有 Python 代码实质上是单线程执行的。但这意味着纯 Python 代码中的多线程只能实现并发，而不能实现真正的并行——同一时刻只有一个线程在执行 Python 代码。

GIL 对 I/O 密集型任务的影响较小，因为在等待 I/O 时 GIL 会被释放，允许其他线程执行。因此 Python 的多线程适合 I/O 密集型应用（如 Web 爬虫、网络服务），但不适合 CPU 密集型计算。对于后者，应使用下面的多进程方案或借助 C 扩展（如 NumPy）在其内部释放 GIL 后进行计算。

### 7.2.2 多进程

要绕过 GIL 实现真正的并行，Python 提供了 `multiprocessing` 模块。每个进程拥有独立的 Python 解释器和独立的内存空间，因此不受 GIL 限制:

```python
import multiprocessing

def worker(name):
    print(f"进程 {name} 正在运行")

p1 = multiprocessing.Process(target=worker, args=("A",))
p2 = multiprocessing.Process(target=worker, args=("B",))

p1.start()
p2.start()

p1.join()
p2.join()
```

进程间不共享内存，因此没有竞态条件的风险。但进程间通信需要通过特殊机制——管道、队列或共享内存——这些机制的开销远大于线程间的直接共享内存访问。具体来说，多进程通信需要:
- **序列化/反序列化**: `multiprocessing.Queue` 和 `Pipe` 使用 `pickle` 将 Python 对象序列化为字节流，接收端再反序列化。复杂对象的序列化开销可能很大。
- **内核态切换**: 进程间管道通信需要系统调用，涉及用户态和内核态的切换。
- **内存拷贝**: 数据需要从发送进程的地址空间拷贝到内核缓冲区，再拷贝到接收进程的地址空间。

权衡法则: 如果任务受 GIL 限制（CPU 密集型），优先使用多进程；如果任务是 I/O 密集型，优先使用多线程。如果任务需要频繁访问大型共享数据结构，线程比进程更方便（因为可以共享内存）。

## 7.3 共享状态的问题: 竞态条件

### 7.3.1 非原子性的本质

考虑以下 Python 代码，两个线程各递增一个共享计数器一次:

```python
import threading

counter = [0]  # 使用列表包装以实现共享可变

def increment():
    temp = counter[0]   # 读
    temp = temp + 1     # 修改
    counter[0] = temp   # 写

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)

t1.start()
t2.start()
t1.join()
t2.join()

print(f"最终计数: {counter[0]}")
```

这不是一个 Python 特有的问题——任何支持共享内存并发的语言都存在这个问题。关键在于 `counter[0] += 1` 不是**原子操作** (atomic operation)。它被编译成三个独立的机器级步骤:

1. **读** (Read, R): 从内存加载 `counter[0]` 的值到 CPU 寄存器
2. **修改** (Modify, M): 在寄存器中执行 ADD 指令将值加 1
3. **写** (Write, W): 将寄存器中的新值通过 STORE 指令写回内存

这一模式称为**读-修改-写** (read-modify-write, RMW) 序列。其核心问题是: 在"读"和"写"之间，另一个线程可能修改了内存中的值，而当前线程对此完全不知情，它仍然基于旧的读取值执行写入。

### 7.3.2 使用 sleep(0) 强制线程切换

GIL 的存在使得简单的计数器测试中竞态条件可能不出现（因为 Python 的某些字节码操作可能在 GIL 释放前恰好完成）。但在多进程或没有 GIL 的语言（如 C++、Java、Go）中，竞态条件是很容易触发的。为了在 Python 中可靠地展示竞态条件，引入 `sleep(0)` 来显式地让出 CPU:

```python
import threading
import time

counter = [0]

def increment():
    for _ in range(100):
        temp = counter[0]
        time.sleep(0)  # 强制 yield GIL
        counter[0] = temp + 1
```

`sleep(0)` 不真正睡眠——它简单地将当前线程置于就绪队列末尾并立即触发一次上下文切换，允许其他线程运行。这使得线程交错几乎在每次迭代都会发生，从而高概率地展示出竞态条件。

### 7.3.3 竞态条件的完整交错分析

**定义**: **竞态条件** (race condition) 发生在程序的行为依赖于两个或多个线程的相对时序（即它们的指令交错方式）时。当结果依赖于"谁先到达某条指令"时，就存在竞态条件。

对于两个线程各执行一次递增操作的基础情形，设初始值 `counter[0] = 0`。每个线程执行三个操作: 读 (R)、修改 (M)、写 (W)。两个线程共有 6 个操作: R0, M0, W0, R1, M1, W1。由于同一线程内部的操作顺序固定（R 必须在 M 之前，M 必须在 W 之前），线程交错必须满足线程内部顺序约束。可能的合法交错总数由以下排列公式给出（其中同一线程的操作视为相同的"类型"但顺序固定）:

$$\frac{6!}{3! \times 3!} = \frac{720}{6 \times 6} = 20 \text{ 种可能的交错}$$

下面枚举产生不同最终结果的代表性交错。

**情形 A: 最终值 counter = 2 (正确结果——两次更新都生效)**

```
时间  | Thread 0              | Thread 1              | counter[0]
------|----------------------|----------------------|-----------
T1    | read counter[0] → 0  |                      | 0
T2    | compute 0 + 1 = 1    |                      | 0
T3    | write 1 → counter    |                      | 1
T4    |                      | read counter[0] → 1  | 1
T5    |                      | compute 1 + 1 = 2    | 1
T6    |                      | write 2 → counter    | 2
```

这是期望的交错。Thread 0 完成完整的 RMW 后，Thread 1 才开始自己的 RMW。Thread 1 读到的值是 Thread 0 写入后的值（1），因此计算结果为 1 + 1 = 2。

**情形 B: 最终值 counter = 1 (经典"丢失更新"问题)**

```
时间  | Thread 0              | Thread 1              | counter[0]
------|----------------------|----------------------|-----------
T1    | read counter[0] → 0  |                      | 0
T2    |                      | read counter[0] → 0  | 0
T3    | compute 0 + 1 = 1    |                      | 0
T4    | write 1 → counter    |                      | 1
T5    |                      | compute 0 + 1 = 1    | 1
T6    |                      | write 1 → counter    | 1
```

两个线程都读到了初始值 0，各自独立计算出 1，然后分别写入 1。Thread 0 的更新被 Thread 1 完全覆盖。**丢失了一次更新**。

**情形 C: 最终值 counter = 1 (对称版本)**

```
时间  | Thread 0              | Thread 1              | counter[0]
------|----------------------|----------------------|-----------
T1    |                      | read counter[0] → 0  | 0
T2    | read counter[0] → 0  |                      | 0
T3    |                      | compute 0 + 1 = 1    | 0
T4    |                      | write 1 → counter    | 1
T5    | compute 0 + 1 = 1    |                      | 1
T6    | write 1 → counter    |                      | 1
```

与情形 B 对称——Thread 1 先读取，Thread 0 后读取，两者都基于初始值 0 计算，最终结果仍为 1。

**情形 D: 最终值 counter = 1 (RMW 交替)**

```
时间  | Thread 0              | Thread 1              | counter[0]
------|----------------------|----------------------|-----------
T1    | read counter[0] → 0  |                      | 0
T2    | compute 0 + 1 = 1    |                      | 0
T3    |                      | read counter[0] → 0  | 0
T4    |                      | compute 0 + 1 = 1    | 0
T5    | write 1 → counter    |                      | 1
T6    |                      | write 1 → counter    | 1
```

在这种交错中，两个线程的"读"和"修改"都发生在任何写操作之前。因此两个线程读取相同值 0，计算相同结果 1，最终写入相同值。

**情形 E: 最终值 counter = 2 (交替但正确)**

```
时间  | Thread 0              | Thread 1              | counter[0]
------|----------------------|----------------------|-----------
T1    | read counter[0] → 0  |                      | 0
T2    |                      | read counter[0] → 0  | 0
T3    |                      | compute 0 + 1 = 1    | 0
T4    |                      | write 1 → counter    | 1
T5    | compute 0 + 1 = 1    |                      | 1
T6    | write 1 → counter    |                      | 1
```

虽然 Thread 0 读到的是 0（在 Thread 1 写入之前），但 Thread 0 的写入在 Thread 1 之后，且在 GIL 下是最后写入者，覆盖为 1。注意这里的关键区别——Thread 0 的 `compute` 在 Thread 1 的 `write` 之后，但 Thread 0 读到的值（在本地变量 `temp` 中）仍然是 0，因此计算结果为 1 而非 2。最终结果仍然是 1。

**命题**: 对于两个线程各递增一次且初始值为 0 的 RMW 问题，最终值只能为 1 或 2，绝不可能为 0。

**证明**: 每个线程将某个读取到的值 $$r$$ 递增为 $$r+1$$ 后写入。每次写入的值都严格大于等于对应的读取值，并且每个写操作至少写入 1（因为读操作返回的值至少为 0，写入值至少为 1）。最后的写入操作决定了最终值。如果两个线程读取到不同的值，则后读取者看到的至少是 1，写入 2。如果两个线程读取到相同的值（都是 0），则各自写入 1，最终值为 1。在所有情况中，最终值绝对不可能为 0。这就证明了至少有一次更新生效。

**扩展**: 更一般地，对于 $$n$$ 个线程各自递增一次的 RMW 问题，最终值在范围 $$[1, n]$$ 内，具体值完全取决于运行时交错的非确定性选择。这清楚地展示了竞态条件的危害——程序的输出不是其输入的函数（违反了引用透明性）。

## 7.4 不需要同步的情形

### 7.4.1 只读数据

如果数据仅被多个线程读取而从不修改，则不需要任何同步。只读共享在多线程中是完全安全的，因为读操作不会改变程序状态:

```python
import threading

SHARED_DATA = [1, 2, 3, 4, 5]  # 只读，线程安全

def process(index):
    value = SHARED_DATA[index]  # 安全: 纯读操作
    print(f"线程处理数据[{index}] = {value}")

threads = [threading.Thread(target=process, args=(i,))
           for i in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

更形式化地说: 如果共享数据是**不可变**的 (immutable) 或者在初始化后不再修改 (effectively immutable)，则对该数据的所有只读访问都是线程安全的，因为任何两个读操作都不互相干扰。这是函数式编程天然适合并发的理论基础。

### 7.4.2 需要深层系统知识的罕见情形

某些特殊情况下，即使存在写入，数据竞争也可能是无害的——但这需要对内存模型、处理器架构和编译器优化有极深的理解，通常应避免:

```python
flag = False
items = [1, 2, 3]

# Thread 0
items.append(4)     # 操作 A: 写入数据
flag = True         # 操作 B: 设置标志

# Thread 1
if flag:            # 操作 C: 检查标志
    print(items)    # 操作 D: 读取数据
```

程序员意图: 操作 A 先于 B，操作 C 先于 D，因此如果 Thread 1 看到 `flag == True`，它一定能看到完整的 `items`（包括新添加的 4）。

**警告: 操作重排序 (Operation Reordering)**

在现代处理器和编译器上，这个假设可能不成立。原因有两层:

1. **编译器重排序**: 编译器在优化代码时可能重排序没有数据依赖关系的操作。因为在编译器看来，操作 A (`items.append`) 和操作 B (`flag = True`) 访问不同的内存位置，没有数据依赖，编译器可能将 B 的机器码放在 A 之前。

2. **处理器重排序**: 即使编译器没有重排序，现代处理器（特别是 x86 的 store buffer 和 ARM/POWER 的宽松内存模型）也可能乱序执行 store 指令。这意味着操作 B 的效果可能先于操作 A 对另一个核心可见。

结果: Thread 1 可能在 `items.append(4)` 的效果对其核心可见之前就看到 `flag == True`，从而打印出不包含 4 的 `items` 列表。

正确的做法: 使用同步原语。例如，使用锁来保护 `flag` 和 `items` 的访问，或者使用原子操作:

```python
import threading

flag = threading.Event()

# Thread 0
items.append(4)
flag.set()  # 释放-获取语义: set() 保证之前的写操作对之后 wait() 的线程可见

# Thread 1
flag.wait()  # 等待 set(), 保证之后能看到之前的所有写入
print(items)  # 安全! 一定能看到 items.append(4)
```

## 7.5 同步数据结构: queue.Queue

Python 的 `queue.Queue` 是一个线程安全的 FIFO 队列实现，内部使用锁和条件变量来保证线程安全。它展示了如何将复杂性封装在数据结构内部，对外提供简单的线程安全 API:

```python
import queue
import threading

q = queue.Queue()

def producer():
    for i in range(5):
        q.put(i)
        print(f"生产: {i}")
    q.put(None)  # 哨兵值 (sentinel) 表示结束

def consumer():
    while True:
        item = q.get()
        if item is None:
            break
        print(f"消费: {item}")
        q.task_done()

t_prod = threading.Thread(target=producer)
t_cons = threading.Thread(target=consumer)

t_prod.start()
t_cons.start()
t_prod.join()
t_cons.join()
```

关键 API:
- `q.put(item)`: 将 `item` 放入队列（如果队列满则阻塞，`Queue` 默认无界）
- `q.get()`: 从队列取出一个 `item`（阻塞直到有可用数据）
- `q.task_done()`: 标记一个队列项已处理完毕，用于配合 `join()` 跟踪未完成的任务数
- `q.join()`: 阻塞直到队列中所有项都已通过 `task_done()` 标记为处理完毕

**守护线程** (daemon thread): 一种在后台运行的线程。关键特性: 当所有非守护线程（主线程 + 所有 `daemon=False` 的线程）结束时，Python 解释器将直接终止所有守护线程而不等待它们完成清理。设置方式: `t.daemon = True`（必须在 `start()` 之前设置）。在生产者-消费者模式中，常将消费者设为守护线程，因为它只是处理数据而不持有必须清理的资源。

**生产者-消费者的形式化模型**: 生产者向队列写入数据，消费者从队列读取数据。队列作为一个有限或无限的缓冲区 (buffer)，解耦了生产者和消费者的处理速度。如果消费者慢于生产者，队列会逐渐填满（或在有界队列中 `put()` 阻塞生产者）；如果生产者慢于消费者，`get()` 会阻塞消费者。这种流量控制机制称为**背压** (backpressure)，是流处理系统中的基础概念。

## 7.6 锁 (Lock / Mutex)

### 7.6.1 基本用法

锁是最基本的同步原语。Python 的 `threading.Lock` 提供互斥访问，确保同一时刻最多只有一个线程持有锁:

```python
import threading

lock = threading.Lock()

# 方式 1: 显式 acquire/release
lock.acquire()
try:
    # --- 临界区开始 ---
    counter[0] += 1
    # --- 临界区结束 ---
finally:
    lock.release()

# 方式 2: with 语句 (推荐)
with lock:
    counter[0] += 1
```

`with lock:` 是上下文管理器语法糖，展开后等价于 `lock.__enter__()` (调用 `acquire()`) 后跟 `try...finally: lock.__exit__()` (调用 `release()`)。这确保即使临界区中发生异常，锁也会被正确释放，避免死锁。

### 7.6.2 临界区

**临界区** (critical section) 是程序中访问共享资源的代码段。其核心约束是: **在同一时刻，最多只能有一个线程执行临界区代码**。这个约束通常被称为**互斥** (mutual exclusion)。

**临界区不变式**: 设一个锁保护共享变量集合 $$S$$。临界区不变式声称: 在任何时刻 $$t$$，最多只有一个线程正在访问 $$S$$ 中的变量。更形式化地:

$$\forall t, |\{thread_i \mid thread_i \text{ 在时刻 } t \text{ 正在访问 } S\}| \leq 1$$

这个不变式保证了 $$S$$ 上的所有操作序列化——任何并发访问都被转化为串行执行，消除了竞态条件的根源。

### 7.6.3 锁保护的计数器完整实现

```python
import threading

counter = [0]
lock = threading.Lock()

def increment():
    with lock:
        temp = counter[0]   # 读 (在临界区内)
        temp = temp + 1     # 修改 (在临界区内)
        counter[0] = temp   # 写 (在临界区内)

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)

t1.start()
t2.start()
t1.join()
t2.join()

print(f"最终计数: {counter[0]}")  # 总是 2
```

### 7.6.4 锁正确性证明

**定理（锁保护计数器的正确性）**: 在使用锁保护且两个线程各对计数器递增一次的情况下，最终计数器值总是为 2，无论线程调度顺序如何。

**证明**: 将每个线程的执行划分为三个连续的阶段:
- **入口协议** (entry protocol): `lock.acquire()`——如果锁被占用则阻塞
- **临界区** (critical section): 读-修改-写 (RMW) 序列
- **出口协议** (exit protocol): `lock.release()`——释放锁并唤醒等待者

锁的抽象状态机有两个状态: `UNLOCKED` 和 `LOCKED`。状态转移规则:
1. `acquire()` 仅在状态为 `UNLOCKED` 时成功，成功后状态变为 `LOCKED`
2. `release()` 仅在状态为 `LOCKED` 时执行，执行后状态变为 `UNLOCKED`

由上述规则可以证明一条核心引理: **互斥引理**——不可能有两个线程同时处于执行临界区的状态。因为要进入临界区，线程必须成功执行 `acquire()`，而 `acquire()` 只在锁处于 `UNLOCKED` 状态时成功，成功的同时锁变为 `LOCKED`。在锁被 `release()` 之前，任何其他线程的 `acquire()` 都会阻塞。因此，两个线程的临界区执行在时间上没有重叠。

由于互斥性，两个临界区一定串行执行。不失一般性，设 Thread 0 先进入临界区:
- Thread 0 读取 `counter[0] = 0`，计算 `0 + 1 = 1`，写入 `counter[0] = 1`
- Thread 0 执行 `release()`，锁变为 `UNLOCKED`
- Thread 1 执行 `acquire()`，锁变为 `LOCKED`，Thread 1 进入临界区
- Thread 1 读取 `counter[0] = 1`（读取到 Thread 0 的写入结果!），计算 `1 + 1 = 2`，写入 `counter[0] = 2`

由于对称性，若 Thread 1 先进入临界区，结果相同。因此最终计数器值总是 2。

**推广（对 k 个线程各递增一次的归纳证明）**: 

对于任意数量的线程 $$n \geq 2$$，每个线程在持有锁时对计数器执行一次递增。由互斥引理，这 $$n$$ 个临界区串行执行。设线程按进入临界区的顺序排列为 $$T_1, T_2, ..., T_n$$。

归纳基础: $$T_1$$ 读到的值为 0，写到 1。
归纳步骤: 假设 $$T_i$$ 读到的值为 $$i-1$$，则 $$T_i$$ 写回值为 $$i$$。$$T_{i+1}$$ 在 $$T_i$$ 的临界区之后进入，因此读到的值是 $$T_i$$ 写入的 $$i$$，计算 $$i+1$$，写入 $$i+1$$。

由归纳法，$$T_n$$ 写入的值为 $$n$$。最终 counter = $$n$$。证毕。

## 7.7 屏障 (Barrier)

### 7.7.1 基本概念

**屏障** (barrier) 是一种同步原语，确保所有参与的线程都到达某个程序点后才能继续执行。屏障将程序执行划分为明确的**阶段** (phase)，保证阶段之间不会重叠。屏障的核心语义是: 当任意线程调用 `barrier.wait()` 时，它阻塞直到所有 `parties` 个线程都调用了 `wait()`。一旦最后一个线程到达，所有被阻塞的线程同时被释放。

```python
import threading

barrier = threading.Barrier(parties=2)  # 2 个参与者

def worker():
    print(f"{threading.current_thread().name}: 阶段 1 开始")
    # 阶段 1 工作 ...
    barrier.wait()
    print(f"{threading.current_thread().name}: 阶段 2 开始")
    # 阶段 2 工作 ...
    barrier.wait()
```

关键 API:
- `threading.Barrier(parties)`: 创建一个需要 `parties` 个参与者才能通过的屏障
- `barrier.wait()`: 阻塞当前线程，直到所有 `parties` 个线程都调用了此方法。返回一个从 0 到 `parties - 1` 的整数，表示当前线程是第几个到达屏障的（可用于分工）

屏障特别适用于**批量同步并行** (Bulk Synchronous Parallel, BSP) 模型，其中计算被组织为一序列的超步 (superstep)，每个超步包含本地计算和全局通信，屏障分隔超步。

### 7.7.2 屏障同步的计数器

下面的例子展示了屏障如何协调两个线程的写入和读取:

```python
import threading

counter = [0, 0]   # 两个线程各自的计数器: counter[0] 由线程 0 写入, counter[1] 由线程 1 写入
barrier = threading.Barrier(parties=2)

def synchronized_counter(my_id):
    other = 1 - my_id
    for step in range(1, 11):  # 10 个时间步
        # 写入阶段: 每个线程给自己的计数器加 1
        counter[my_id] += 1
        barrier.wait()

        # 读取阶段: 每个线程读取对方的计数器并验证
        assert counter[other] == step, \
            f"阶段不变式被违反! 线程 {my_id} 在第 {step} 步读到 counter[{other}] = {counter[other]}"
        barrier.wait()

t1 = threading.Thread(target=synchronized_counter, args=(0,))
t2 = threading.Thread(target=synchronized_counter, args=(1,))

t1.start()
t2.start()
t1.join()
t2.join()
print("所有断言通过! 屏障同步保证了阶段不变式。")
```

### 7.7.3 屏障正确性证明

**定义（阶段分离不变式）**: 对于任意步数 $$k \geq 1$$，在任何时刻，不存在一个线程在步骤 $$k$$ 的读取阶段而另一个线程仍在步骤 $$k$$ 的写入阶段。

**定理（屏障同步计数器正确性）**: 对于上述程序，在第 $$k$$ 步的读取阶段，`counter[0] = counter[1] = k` 成立。

**证明（对步数 $$k$$ 的归纳法）**:

**基例 $$k = 1$$**:

在步骤 1 的写入阶段:
- Thread 0 执行 `counter[0] += 1`，使 `counter[0] = 1`
- Thread 1 执行 `counter[1] += 1`，使 `counter[1] = 1`

两条线程各自到达第一个 `barrier.wait()`。屏障语义确保: 只有当两条线程都到达屏障后，任何一条线程才能从 `wait()` 返回并进入读取阶段。因此，当线程进入步骤 1 的读取阶段时，两条线程都已完成了步骤 1 的写入。

此时:
- `counter[0] = 1`（Thread 0 已写入）
- `counter[1] = 1`（Thread 1 已写入）
- 断言 `counter[other] == 1` 对两条线程都成立

**归纳假设**: 假设在第 $$k$$ 步的读取阶段，`counter[0] = counter[1] = k` 成立，且所有线程都已通过第 $$k$$ 步的屏障进入读取阶段。

**归纳步骤（证明第 $$k+1$$ 步）**:

两条线程继续执行，经过第二个屏障后进入第 $$k+1$$ 步的写入阶段:
- 屏障确保两条线程都完成了第 $$k$$ 步的读取后才进入下一步
- Thread 0 执行 `counter[0] += 1`，将 `counter[0]` 从 $$k$$ 更新为 $$k+1$$
- Thread 1 执行 `counter[1] += 1`，将 `counter[1]` 从 $$k$$ 更新为 $$k+1$$

两条线程各自到达第 $$k+1$$ 步的写入屏障。同样，屏障确保两条线程都完成写入后才释放它们进入读取阶段。

因此，在第 $$k+1$$ 步的读取阶段:
- `counter[0] = k + 1`
- `counter[1] = k + 1`
- 断言 `counter[other] == k+1` 对两条线程都成立

由归纳法原理，命题对所有步数 $$k \geq 1$$ 成立。证毕。

**核心洞察**: 屏障通过强制所有线程同步在两个关键点（写入后和读取后），保证了数据更新的全序可见性。在写入屏障处，它保证所有线程的写入对该步已完成；在读取屏障处，它保证所有线程都已观察到对方的写入。这种两阶段同步模式是许多并行算法的基石。

## 7.8 消息传递 (Message Passing)

**消息传递** (message passing) 是另一种并发模型，线程/进程间不直接共享内存，而是通过发送和接收消息来通信。这是 Go 语言和 Erlang 等语言的核心并发哲学——"不通过共享内存来通信，而通过通信来共享内存" (Don't communicate by sharing memory; share memory by communicating)。

### 7.8.1 管道 (Pipe)

`multiprocessing.Pipe` 创建一个连接两个进程的双向或单向通信通道:

```python
import multiprocessing

parent_conn, child_conn = multiprocessing.Pipe(duplex=True)

def sender(conn):
    for i in range(5):
        conn.send(f"消息 {i}")
    conn.send(None)  # 哨兵

def receiver(conn):
    while True:
        msg = conn.recv()
        if msg is None:
            break
        print(f"接收: {msg}")

p1 = multiprocessing.Process(target=sender, args=(child_conn,))
p2 = multiprocessing.Process(target=receiver, args=(parent_conn,))

p1.start()
p2.start()
p1.join()
p2.join()
```

关键 API:
- `multiprocessing.Pipe(duplex=True)`: 创建管道。`duplex=True` 时返回 `(conn1, conn2)`，两端都可以 `send()` 和 `recv()`；`duplex=False` 时返回 `(recv_conn, send_conn)`，一端只接收，一端只发送
- `conn.send(obj)`: 发送对象。被发送的对象通过 `pickle` 序列化为字节流，然后写入管道。如果管道内部缓冲区已满，`send()` 会阻塞直到有空间
- `conn.recv()`: 接收对象。如果管道为空，`recv()` 会阻塞直到有数据到达。接收到的字节流通过 `pickle` 反序列化恢复为 Python 对象

**阻塞语义警告**: `send()` 和 `recv()` 都是阻塞的。当两个进程都试图先 `recv()` 时，会发生经典的双向管道死锁（见 7.10.3 节分析）。

对于进程间同步，`multiprocessing` 模块还提供了 `Lock`、`Event`、`Condition`、`Semaphore` 和 `Barrier` 等同步原语，它们的 API 与 `threading` 模块中的对应物一致，但实现为跨进程的原语，通过底层的命名信号量或共享内存实现。

### 7.8.2 基于管道的生产者-消费者

```python
import multiprocessing

def producer(conn):
    for i in range(10):
        conn.send(i)
    conn.send(None)  # 哨兵表示结束

def consumer(conn):
    while True:
        item = conn.recv()
        if item is None:
            break
        print(f"消费: {item}")

recv_conn, send_conn = multiprocessing.Pipe(duplex=False)

p_prod = multiprocessing.Process(target=producer, args=(send_conn,))
p_cons = multiprocessing.Process(target=consumer, args=(recv_conn,))

p_prod.start()
p_cons.start()
p_prod.join()
p_cons.join()
```

注意这里使用了单向管道 (`duplex=False`)，`send_conn` 只能发送，`recv_conn` 只能接收。这从设计层面避免了双向通信中可能的死锁风险。

## 7.9 应用示例

### 7.9.1 Web 爬虫

Web 爬虫 (web crawler) 是并发编程的经典应用场景: 从一些种子 URL 开始，并发下载页面，提取页面中的超链接，将新链接加入待爬取队列。

```python
import threading
import queue

class WebCrawler:
    def __init__(self, seed_url, num_threads=4):
        self.work_queue = queue.Queue()
        self.work_queue.put(seed_url)
        self.seen = set()                # 记录已见过的 URL
        self.seen_lock = threading.Lock() # 保护 seen 集合

    def already_seen(self, url):
        """原子化的"检查并添加"操作——线程安全的去重"""
        with self.seen_lock:
            if url in self.seen:
                return True
            self.seen.add(url)
            return False

    def crawl(self):
        """工作线程的主循环"""
        while True:
            try:
                url = self.work_queue.get(timeout=1)
            except queue.Empty:
                return  # 超时未拿到任务，退出
            try:
                page = self.download(url)
                links = self.extract_links(page)
                for link in links:
                    if not self.already_seen(link):
                        self.work_queue.put(link)
            finally:
                self.work_queue.task_done()

    def download(self, url):
        # 实际页面下载逻辑
        pass

    def extract_links(self, page):
        # 实际链接提取逻辑
        pass

    def start(self, num_threads=4):
        threads = []
        for _ in range(num_threads):
            t = threading.Thread(target=self.crawl)
            t.start()
            threads.append(t)
        self.work_queue.join()  # 等待所有任务完成
        for t in threads:
            t.join()
```

**`already_seen` 的正确性证明**:

`already_seen` 方法将"检查"和"添加"两个操作合并为一个受同一锁保护的原子操作。我们来证明这种设计正确地防止了重复 URL 处理。

**错误实现（分裂锁版本）**:

```python
# 错误! "检查"和"添加"各自加锁，但不是同一个原子整体
def is_seen(self, url):
    with self.seen_lock:
        return url in self.seen     # 临界区 A

def add_seen(self, url):
    with self.seen_lock:
        self.seen.add(url)          # 临界区 B

# 在 crawl() 中的用法:
for link in links:
    if not self.is_seen(link):      # 锁获取→检查→锁释放
        self.add_seen(link)         # 锁获取→添加→锁释放
        self.work_queue.put(link)
```

**失败的交错分析**:

```
时间  | Thread A (检查 url_X)    | Thread B (检查 url_X)    | seen 集合
------|--------------------------|--------------------------|-----------
T1    | is_seen: acquire lock    |                          |
T2    | url_X not in seen → False|                          | {}
T3    | is_seen: release lock    |                          |
T4    |                          | is_seen: acquire lock    |
T5    |                          | url_X not in seen → False| {}
T6    |                          | is_seen: release lock    |
T7    | add_seen: acquire lock   |                          |
T8    | seen.add(url_X)          |                          | {url_X}
T9    | add_seen: release lock   |                          |
T10   |                          | add_seen: acquire lock   |
T11   |                          | seen.add(url_X) (重复!)  | {url_X}
T12   |                          | add_seen: release lock   |
T13   | work_queue.put(url_X)    |                          |
T14   |                          | work_queue.put(url_X)    | 重复入队!
```

在 T2 和 T5 时刻，两个线程在各自的临界区 A 中独立检查，都得到 `url_X` 不在 `seen` 中。然后，它们各自在临界区 B 中（T7-T9 和 T10-T12）分别添加 `url_X`。由于临界区 A 和 B 之间没有锁保护，其他线程可以在间隙中插入，导致重复检查和重复添加。

在正确的实现中，`already_seen` 将检查（T2 对应的操作）和添加（T8 对应的操作）放在同一个 `with self.seen_lock:` 块内。这意味着在 T2 和 T8 之间，没有其他线程能获取 `seen_lock`。因此，一旦一个线程在锁内确认 `url_X` 不在 `seen` 中并添加了它，任何后续线程在检查时必然发现 `url_X` 已在 `seen` 中，从而正确地跳过。

**形式化验证**: 设 `seen` 初始为空。在任意执行中，对于任意 URL $$u$$，操作序列为:
- 第一个遇到 $$u$$ 的线程: `acquire → (u ∉ seen) → add(u) → release`。此时 `seen` 变为包含 $$u$$
- 第二个及之后遇到 $$u$$ 的线程: `acquire → (u ∈ seen) → return True → release`。不执行添加

由于检查和添加是原子的，不存在"两个线程都看到 $$u$$ 不在 `seen` 中"的状态。因此每个 URL 被添加至多一次至工作队列，从而保证每个 URL 被爬取至多一次。证毕。

### 7.9.2 粒子模拟器

考虑一个包含 $$N$$ 个粒子的物理模拟。每个粒子在二维空间中由位置向量 $$\vec{p}$$、速度向量 $$\vec{v}$$ 和加速度向量 $$\vec{a}$$ 描述。模拟在每个离散时间步 $$\Delta t$$ 上通过以下数值积分更新状态:

$$\vec{a}_i^{(t)} = \frac{1}{m_i} \sum_{j \neq i} \vec{F}_{ij}^{(t)}$$

$$\vec{v}_i^{(t+1)} = \vec{v}_i^{(t)} + \vec{a}_i^{(t)} \cdot \Delta t$$

$$\vec{p}_i^{(t+1)} = \vec{p}_i^{(t)} + \vec{v}_i^{(t+1)} \cdot \Delta t$$

其中 $$\vec{F}_{ij}$$ 是粒子 $$j$$ 对粒子 $$i$$ 施加的力（例如引力 $$F = G m_i m_j / r^2$$ 或静电库仑力）。

#### 四阶段算法（屏障版本，线程并行）

每个时间步包含四个阶段，由两个屏障分隔:

**阶段 1——计算力**: 每个线程处理一部分粒子，计算每个被分配粒子与所有其他粒子之间的力，累积到加速度中
**屏障 1**: 确保所有力的计算完成后任何线程才能更新速度
**阶段 3——更新速度与位置**: 使用新计算的加速度更新速度和位置
**屏障 2**: 确保所有粒子的状态更新完成后才能进入下一时间步

```python
import threading

class ParticleSimulator:
    def __init__(self, particles, dt, num_threads=4):
        self.particles = particles    # 粒子列表
        self.dt = dt                  # 时间步长
        self.num_threads = num_threads
        self.force_barrier = threading.Barrier(num_threads)
        self.update_barrier = threading.Barrier(num_threads)

    def worker(self, thread_id, start_idx, end_idx, steps):
        """工作线程: 处理粒子索引 [start_idx, end_idx)"""
        N = len(self.particles)

        for step in range(steps):
            # 阶段 1: 计算加速度 (力 / 质量)
            for i in range(start_idx, end_idx):
                self.particles[i].accel = 0.0  # 重置
                for j in range(N):
                    if i != j:
                        force = self.compute_force(
                            self.particles[i], self.particles[j])
                        self.particles[i].accel += force / self.particles[i].mass

            self.force_barrier.wait()  # 屏障 1: 同步力计算

            # 阶段 3: 更新速度 (用新加速度) 和位置 (用新速度)
            for i in range(start_idx, end_idx):
                self.particles[i].velocity += \
                    self.particles[i].accel * self.dt
                self.particles[i].position += \
                    self.particles[i].velocity * self.dt

            self.update_barrier.wait()  # 屏障 2: 同步状态更新
```

屏障在此处的语义分析:

1. **屏障 1** 确保阶段分离不变式: 当线程进入阶段 3 时，所有线程都已完成阶段 1。如果缺少此屏障，一个快速的线程可能在另一个线程还在计算粒子间作用力时就更新速度和位置，导致该时间步中使用的是不完整的（部分累积好的）加速度——物理上毫无意义。

2. **屏障 2** 确保时间步不变式: 当线程进入下一时间步的阶段 1 时，所有线程都已完成当前时间步的阶段 3。如果缺少此屏障，一个线程可能基于部分更新的粒子位置计算力，导致力的计算基于混合了当前步和下一步的位置——破坏了数值积分的正确性。

#### 环形管线（消息传递版本，进程并行）

消息传递版本将粒子分配给多个进程，这些进程按环形拓扑组织。数据沿环流动，每个进程处理自己分配的粒子，然后将结果传递给下一个进程:

```python
import multiprocessing

class RingPipelineSimulator:
    def __init__(self, particles, num_processes):
        self.local_particles = particles  # 分配给进程的粒子
        self.num_procs = num_processes

    def create_ring(self):
        """创建环形管道网络: 进程 i 连接进程 (i+1) % N"""
        pipes = []
        for i in range(self.num_procs):
            parent, child = multiprocessing.Pipe(duplex=True)
            pipes.append((parent, child))
        return pipes

    def worker(self, proc_id, conn_left, conn_right, steps):
        """conn_left: 从上一进程接收, conn_right: 发送给下一进程"""
        for step in range(steps):
            # 接收来自上一进程的数据
            received_data = conn_left.recv()

            # 本地计算: 更新自己的粒子 + 处理接收到的数据
            local_result = self.compute_local(proc_id, received_data)

            # 发送结果给下一进程
            conn_right.send(local_result)
```

环形管线的关键优势: 完全消除了共享内存，从而从根本上消除了竞态条件的可能性。代价是通信开销——粒子数据需要序列化并通过管道传输。对于粒子间相互作用力需要全部信息的问题，环形拓扑要求所有数据绕环一周，通信量较大。

## 7.10 同步陷阱

同步编程的陷阱多种多样，本节逐一分析三种最常见的错误模式。

### 7.10.1 同步不足 (Under-Synchronization)

同步不足发生在需要原子执行的复合操作被拆分为多个分别加锁的步骤时。Web 爬虫分裂锁的例子已经在 7.9.1 节中详细分析过。另一个经典例子:

```python
# 错误! count 的每次操作都加了锁，但复合操作不是原子的
with lock:
    c = counter[0]
# lock 在此释放!
with lock:
    counter[0] = c + 1
```

即使每次单独的访问都加了锁，读和写之间的间隙使得其他线程可以插入它们的递增操作。这两个锁保护的临界区是两个不同的原子块，而不是一个。关键教训: **锁保护的粒度必须匹配需要原子执行的粒度**。

### 7.10.2 过度同步 (Over-Synchronization)

过度同步会降低并行度，甚至导致**饥饿** (starvation)——一个线程长时间得不到执行的机会:

```python
# 反模式: 在持有锁时执行长耗时操作
with lock:
    item = queue.pop()
    process_item(item)  # 耗时操作! 在持有锁时执行
```

`process_item` 是一个潜在的长耗时操作（例如磁盘 I/O 或复杂计算）。在持有锁时执行它意味着其他所有需要该锁的线程都被阻塞，即使它们只需要一瞬间就能完成操作。这严重降低了并发性，可能将程序退化为单线程。

**正确做法**: 尽可能缩小临界区的范围——只保护共享状态的访问，将处理放在锁外:

```python
with lock:
    item = queue.pop()
# 锁在此释放，耗时操作在锁外执行
process_item(item)
```

这被称为**临界区最小化原则**: 临界区应该只包含对共享变量的直接操作，不应包含任何与共享状态无关的计算。

### 7.10.3 死锁 (Deadlock)

**死锁** (deadlock) 发生在两个或多个线程互相等待对方持有的资源时，形成循环等待 (circular wait)，导致所有参与的线程永久阻塞。

#### 双向管道死锁示例

```python
import multiprocessing

def process_a(conn):
    msg = conn.recv()            # A 先等待从 B 接收
    conn.send(f"A 收到: {msg}")  # 然后发送

def process_b(conn):
    msg = conn.recv()            # B 也先等待从 A 接收
    conn.send(f"B 收到: {msg}")  # 然后发送

conn_a, conn_b = multiprocessing.Pipe(duplex=True)

p_a = multiprocessing.Process(target=process_a, args=(conn_a,))
p_b = multiprocessing.Process(target=process_b, args=(conn_b,))

p_a.start()
p_b.start()
p_a.join()
p_b.join()  # 死锁! 永远不会结束
```

#### 死锁的资源分配图

用有向图建模死锁: 进程节点和资源节点。请求边 (request edge) 从进程指向资源，表示进程在等待该资源。分配边 (assignment edge) 从资源指向进程，表示该资源已分配给该进程。图中存在循环 (cycle) 时即发生活锁或将要发生死锁:

```
    请求 recv                  请求 recv
  P_a ─────────> 管道 ────────────> P_b
                  ^                 ^
                  |    循环等待      |
                  |<───────────────>|
```

**详细分析**: P_a 调用 `recv()` 时，它将阻塞直到管道中有数据。管道中的数据只能来自 P_b 的 `send()`。但是 P_b 也在 `recv()` 处阻塞，等待 P_a 发送数据。两个进程在各自的第一行代码就互相阻塞，永远无法到达能解除对方阻塞的 `send()`。

#### 死锁的四个必要条件 (Coffman et al., 1971)

1. **互斥** (Mutual Exclusion): 资源每次最多只能被一个进程使用
2. **持有并等待** (Hold and Wait): 进程已经持有某些资源，同时等待获取另外的资源
3. **不可抢占** (No Preemption): 资源只能由持有者自愿释放，不能被外部强制剥夺
4. **循环等待** (Circular Wait): 存在进程集合 $$\{P_0, P_1, ..., P_n\}$$ 使得 $$P_0$$ 等待 $$P_1$$ 占有的资源，$$P_1$$ 等待 $$P_2$$ 占有的资源，...，$$P_n$$ 等待 $$P_0$$ 占有的资源

破除四个条件中的任何一个即可防止死锁。对于上述管道死锁，破除循环等待即可。

#### 修正: 确保一致的通信顺序

```python
def process_a(conn):
    conn.send("Hello from A")    # 先发送!
    msg = conn.recv()            # 再接收
    print(f"A 收到: {msg}")

def process_b(conn):
    msg = conn.recv()            # 先接收
    print(f"B 收到: {msg}")
    conn.send("Hello from B")    # 再发送
```

**无死锁证明**:

设两个进程启动后。考虑执行顺序:

**情形 1——P_a 先到达 send**: P_a 发送消息后，管道中有待接收的数据。P_a 随后在 `recv()` 处阻塞，等待 P_b 的响应。P_b 到达 `recv()`，读取到 P_a 发送的消息，随后执行 `send()` 发送响应，解除 P_a 的阻塞。P_a 从 `recv()` 返回后继续执行。两个进程都能完成。无死锁。

**情形 2——P_b 先到达 recv**: P_b 在 `recv()` 处阻塞，等待管道中有数据。P_a 随后到达 `send()`，发送消息到管道。此消息解除 P_b 在 `recv()` 处的阻塞。P_b 继续执行到 `send()`，发送响应，解除 P_a 在 `recv()` 处的阻塞。两个进程都能完成。无死锁。

在所有可能的执行顺序下，都没有形成循环等待——因为 P_a 总是先 `send` 后 `recv`，而 P_b 总是先 `recv` 后 `send`。不会出现双方同时 `recv` 的状态。证毕。

**更一般的解决方案: 一致锁排序 (Consistent Lock Ordering)**

对于涉及多个锁的死锁问题，一致锁排序是标准解法。规则: 所有线程必须始终以相同的全局顺序获取锁。如果存在锁 $$L_1, L_2, ..., L_k$$，规定所有线程必须按索引递增顺序获取锁（即先获取 $$L_1$$ 再获取 $$L_2$$，以此类推）。这从根本上消除了循环等待条件，因为不可能存在一个线程持有 $$L_i$$ 等待 $$L_j$$ 而另一个线程持有 $$L_j$$ 等待 $$L_i$$（两者都违反了一致的顺序）。

## 7.11 异步任务 (Asynchronous Tasks in C++)

C++ 11 引入的 `std::async` 和 `std::future` 提供了一种高级的基于任务的 (task-based) 并行模型。与底层线程管理不同，`std::async` 将计算提交给运行时——程序员指定"做什么"而非"怎么做"。

```cpp
#include <future>
#include <iostream>

int compute(int x) {
    return x * x;
}

int main() {
    // 异步启动 compute(42)——返回 future 作为结果的"占位符"
    std::future<int> result = std::async(std::launch::async, compute, 42);

    // 在此期间可以执行其他有用的工作 ...

    // 获取结果（如果计算尚未完成则阻塞）
    int value = result.get();
    std::cout << "结果是: " << value << std::endl;  // 输出 1764
    return 0;
}
```

关键 API:
- `std::async(launch_policy, callable, args...)`: 异步执行可调用对象，返回一个 `std::future<T>` 对象
- `std::future<T>::get()`: 获取结果。如果计算尚未完成，调用线程将阻塞直到结果就绪。**重要**: `get()` 只能调用**一次**——调用后 future 进入无效状态
- `std::future<T>::wait()`: 等待结果就绪但不获取。可以多次调用
- `std::launch::async`: 强制在新线程中执行（立即启动）
- `std::launch::deferred`: 延迟求值——任务在第一次调用 `get()` 或 `wait()` 时才在调用者线程中执行

### 7.11.1 异步 Fibonacci 与任务爆炸

斐波那契数列的递归定义:
$$F(0) = 0, \quad F(1) = 1$$
$$F(n) = F(n-1) + F(n-2) \quad (n \geq 2)$$

直观上，每个 $$F(n)$$ 的计算可以分解为两个独立的子问题 $$F(n-1)$$ 和 $$F(n-2)$$，似乎适合并行化。天真地用 `std::async` 实现:

```cpp
int async_fib(int n) {
    if (n <= 1) return n;

    auto fut1 = std::async(std::launch::async, async_fib, n-1);
    auto fut2 = std::async(std::launch::async, async_fib, n-2);

    return fut1.get() + fut2.get();
}
```

**问题诊断**: 每次调用递归地创建两个异步任务。任务总数 $$T(n)$$ 满足递推关系:

$$T(0) = 0, \quad T(1) = 0$$
$$T(n) = 1 + T(n-1) + T(n-2) \quad (n \geq 2)$$

这本质上就是斐波那契数自身: $$T(n) = F(n+2) - 1$$。任务数以指数级速度增长，增长率受黄金比例 $$\phi = \frac{1+\sqrt{5}}{2} \approx 1.618$$ 支配。对于 $$n = 30$$，任务数约为 $$F(32) \approx 2.1 \times 10^6$$——约 210 万个任务。

这种指数爆炸是灾难性的。实验显示，无限制版异步 Fibonacci 的耗时约为顺序版的 **4000 倍**。原因: 创建每个异步任务需要分配 `std::future` 状态、可能的线程创建、上下文切换、最终结果传递等开销，而每个任务的"有用计算"仅仅是两个整数的加法。开销/有效工作比趋于无穷大。

### 7.11.2 任务限制策略与精细分析

解决方案是限制同时活跃的异步任务数量。核心思想: 维护一个追踪变量 `tasks`，当 `tasks` 超过阈值 `max_tasks` 时回退到顺序执行:

```cpp
int async_fib(int n, int tasks, int max_tasks) {
    if (n <= 1) return n;

    if (2 * tasks > max_tasks) {
        // 回退到顺序执行: 足够多的任务, 不再创建新任务
        return async_fib(n-1, tasks, max_tasks)
             + async_fib(n-2, tasks, max_tasks);
    }

    // 并行: 创建两个异步子任务
    auto fut1 = std::async(std::launch::async,
                           async_fib, n-1, tasks, max_tasks);
    auto fut2 = std::async(std::launch::async,
                           async_fib, n-2, tasks, max_tasks);
    return fut1.get() + fut2.get();
}
```

#### 追踪量 `2 * tasks` 的推导

在递归树的每一层，一个调用如果决定并行化，它会创建两个异步任务。初始调用 `async_fib(n, 1, max_tasks)` 中 `tasks = 1`（表示当前这个任务）。如果 `2 * 1 <= max_tasks`，则并行创建两个子任务。

现在，`tasks` 参数被原样传递给子任务。这意味着在递归树的第 $$d$$ 层，每个任务携带的 `tasks` 值仍然是 1。但每个任务会创建两个子任务，因此第 $$d+1$$ 层的任务数是第 $$d$$ 层的 2 倍。`2 * tasks` 是每个任务对"如果分裂将产生的总任务数"的估计——当前任务自身 (`tasks`) 加上两个子任务（各携带 `tasks`），总计约为 `2 * tasks`（近似）。

### async_fib(5, 1, 4) 的任务树分析

以 `max_tasks = 4` 为例，追踪 `async_fib(5, 1, 4)` 的执行:

```
                         fib(5) [tasks=1]
                         2*1=2≤4: 并行分裂
                       /                    \
            fib(4) [tasks=1]          fib(3) [tasks=1]
            2*1=2≤4: 并行             2*1=2≤4: 并行
           /            \              /            \
   fib(3) [t=1]   fib(2) [t=1]  fib(2) [t=1]  fib(1) [t=1]
   2*1=2≤4: 并行  2*1=2≤4: 并行 2*1=2≤4: 并行  n≤1: 返回1
     /    \        /    \        /    \
fib(2) fib(1)  fib(1) fib(0)  fib(1) fib(0)
[t=1]  [t=1]   [t=1]  [t=1]   [t=1]  [t=1]

后续所有 2*1=2≤4 条件继续满足，
但 n 逐渐减小直至 n≤1 触底返回
```

当 `max_tasks = 4` 且 `n = 5` 时，`2 * 1 = 2` 始终满足并行条件，因此二叉递归树的前 3 层都被并行化。在第 3 层（叶子层），`n` 衰减到 1 或 0，触底返回。由于 `max_tasks = 4` 但 `tasks` 始终为 1，实际上任务限制没有触发——二叉树的每个内部节点都并行化了。实际创建的异步任务数约为 $$2^{3} - 1 = 7$$ 个（高度为 3 的完全二叉树的内节点数）。

当 `max_tasks = 2` 时:
- `fib(5)`: `2 * 1 = 2 <= 2`，并行分裂
- `fib(4)`: `2 * 1 = 2 <= 2`，并行分裂
- `fib(3)`: `2 * 1 = 2 <= 2`，并行分裂
- ...
此时所有的并行决策都相同，因为 `tasks` 始终等于 1

当 `max_tasks = 1` 时:
- `fib(5)`: `2 * 1 = 2 > 1`，顺序执行! 回退到顺序递归

实验表明，`max_tasks` 的最佳值取决于具体问题。对于 Fibonacci，较小的值（如 4 到 16）通常最佳，因为每个任务的粒度太小了（两个整数加法），过多的并行只会增加调度开销。

### 7.11.3 异步快速排序与 Amdahl 定律

快速排序的平均时间复杂度为 $$O(n \log n)$$，其分治结构天然适合任务并行:

```cpp
template <typename It>
void async_quicksort(It begin, It end, int tasks, int max_tasks) {
    if (begin >= end) return;

    // Partition: 将元素按 pivot 分成两半
    auto pivot = *begin;
    auto mid = std::partition(begin, end,
                              [=](const auto& x) { return x < pivot; });

    if (2 * tasks > max_tasks) {
        // 顺序执行左右两半
        async_quicksort(begin, mid, tasks, max_tasks);
        async_quicksort(mid + 1, end, tasks, max_tasks);
    } else {
        // 并行执行左右两半
        auto fut1 = std::async(std::launch::async, async_quicksort<It>,
                               begin, mid, tasks, max_tasks);
        auto fut2 = std::async(std::launch::async, async_quicksort<It>,
                               mid + 1, end, tasks, max_tasks);
        fut1.get();
        fut2.get();
    }
}
```

实验结果显示，当 `max_tasks = 128` 时，并行快速排序相对于顺序版本实现了约 $$2.7\times$$ 的加速比。

#### Amdahl 定律: 理论加速比上界

**Amdahl 定律** (Amdahl's Law) 描述了并行程序的理论加速比上限。设程序总工作量为 1，其中分数 $$p$$ 可被并行化，分数 $$(1-p)$$ 必须顺序执行。使用 $$n$$ 个处理器时，并行部分的执行时间为 $$p/n$$，顺序部分的执行时间仍为 $$(1-p)$$。因此总加速比为:

$$S(n) = \frac{1}{(1-p) + \frac{p}{n}}$$

**极限分析**: 当处理器数量趋于无穷时:

$$\lim_{n \to \infty} S(n) = \frac{1}{1-p}$$

这是 Amdahl 定律最著名的推论: **顺序部分决定了加速比的上限**。如果一个程序只有 90% 可并行化 ($$p = 0.9$$)，那么无论使用多少处理器，加速比不会超过 $$1/0.1 = 10$$ 倍。

**快速排序的 Amdahl 分析**:

快速排序中不可避免的顺序部分包括:
1. 主线程的初始设置、partition 操作和线程管理（约占总时间的 5%~8%）
2. 子数组递归到底部后的顺序排序——在数组中只有少量元素时，顺序排序比创建异步任务更高效
3. 任务调度、同步和 future 管理开销（约占总时间的 5%~8%）

一个现实的估计可以假设并行化比例 $$p \approx 0.85$$（即 15% 的顺序部分）。Amdahl 定律给出的理论上限为:

$$S_\infty = \frac{1}{1 - 0.85} = \frac{1}{0.15} \approx 6.67$$

实际观察到的 2.7x 加速比远低于这个上限，说明还有额外因素在降低并行效率:
- **缓存失效** (cache miss): 多线程共享 L3 缓存，如果线程在不同的核心上运行且频繁修改数据，缓存一致协议会产生大量流量
- **伪共享** (false sharing): 两个线程修改位于同一缓存行中的不同变量，导致缓存行在核心间不断反弹
- **任务粒度过小**: 即使有了 128 的任务限制，数组很小时创建异步任务的开销仍大于顺序排序的收益
- **负载不均衡**: partition 操作可能产生严重不均衡的分区，导致某些线程的任务远比其他的大

### 7.11.4 Monte Carlo 方法计算 π

Monte Carlo 方法通过随机采样来估计确定性量。对于 π 的估计，经典的"投针法"等效方案如下:

**推导**: 在单位正方形 $$[0,1] \times [0,1]$$ 中随机生成点 $$(x, y)$$，其中 $$x, y \sim U(0,1)$$。点落在内切四分之一单位圆内的条件是:

$$x^2 + y^2 \leq 1$$

四分之一单位圆的面积为:

$$A_{\text{四分之一圆}} = \frac{\pi \cdot r^2}{4} = \frac{\pi}{4}$$

单位正方形的面积为 1。由均匀采样的几何概率，点落入圆内的概率等于圆面积与正方形面积之比:

$$P(\text{点在圆内}) = \frac{\pi/4}{1} = \frac{\pi}{4}$$

设进行了 $$N$$ 次采样，其中 $$N_{in}$$ 个点落在圆内。由大数定律 (Law of Large Numbers)，当 $$N$$ 充分大时:

$$\frac{N_{in}}{N} \approx \frac{\pi}{4}$$

因此 π 的估计为:

$$\pi \approx 4 \times \frac{N_{in}}{N}$$

**统计误差分析**: Monte Carlo 估计的标准误差由二项分布的性质决定。每个采样点是一个伯努利试验（成功概率 $$p = \pi/4$$）。对于 $$N$$ 个独立试验，$$N_{in}$$ 的方差为 $$N p (1-p)$$。通过 delta 方法，$$\hat{\pi} = 4 N_{in} / N$$ 的方差为:

$$\text{Var}(\hat{\pi}) = \frac{16}{N^2} \cdot N p (1-p) = \frac{16 p (1-p)}{N}$$

标准误差为:

$$\sigma_{\hat{\pi}} = \frac{4\sqrt{p(1-p)}}{\sqrt{N}} = O\left(\frac{1}{\sqrt{n}}\right)$$

这是一个重要的结论: **Monte Carlo 方法的误差以 $$O(1/\sqrt{n})$$ 速率收敛**。这意味着要获得额外一位有效数字的精度（误差缩小到 1/10），需要将采样点数增加到原来的 $$10^2 = 100$$ 倍。

```cpp
#include <random>
#include <future>
#include <vector>

double monte_carlo_pi_parallel(long long samples, int num_workers) {
    long long per_worker = samples / num_workers;
    std::vector<std::future<long long>> futures;

    for (int w = 0; w < num_workers; ++w) {
        futures.push_back(std::async(std::launch::async, [=]() {
            // 每个 worker 有自己的随机数生成器 (避免共享状态的竞态)
            std::random_device rd;
            std::mt19937 gen(rd());
            std::uniform_real_distribution<> dist(0.0, 1.0);

            long long inside = 0;
            for (long long i = 0; i < per_worker; ++i) {
                double x = dist(gen);
                double y = dist(gen);
                if (x * x + y * y <= 1.0) ++inside;
            }
            return inside;
        }));
    }

    long long total_inside = 0;
    for (auto& fut : futures) {
        total_inside += fut.get();
    }
    return 4.0 * total_inside / samples;
}
```

**性能对比** (以 $$10^8$$ 个样本为例):

| 配置 | 耗时 | 加速比 | 效率 (加速比 / workers) |
|------|------|--------|------------------------|
| 顺序 | ~1.86s | 1.00x | -- |
| 2 workers | ~0.95s | 1.96x | 98% |
| 4 workers | ~0.52s | 3.58x | 89% |

Monte Carlo 方法是典型的"尴尬并行" (embarrassingly parallel) 问题。每个 worker 完全独立: 各自拥有独立的随机数生成器，各自计算部分样本的圆内命中数，没有共享状态也不需要通信。因此它接近 Amdahl 定律的理想情况——$$p \approx 1.0$$，理论上限 $$S_\infty \to \infty$$。实际加速比低于理想值主要是因为:
- 主线程聚合结果和创建/管理 future 的开销（顺序部分 $$1 - p$$ 的小但非零）
- 内存带宽限制——多个线程同时从主存读取随机数可能受限于内存带宽

## 7.12 本章小结

并发编程是现代程序员不可或缺的技能，因为硬件演进已从提升单核频率转向增加核心数量。本章涵盖了从基础概念到高级异步编程的全部内容:

1. **并发 vs 并行**: 并发是逻辑上的任务重叠（交错执行），并行是物理上的同时执行（多核）。理解这一区分是正确选择并发模型的前提
2. **竞态条件**: 没有同步保护的共享可变状态导致非确定性行为。两个线程各递增一次计数器，在不加锁时结果可能是 1（丢失更新）或 2（两次更新都生效）。交错的总可能性数量随线程数和指令数呈组合爆炸式增长
3. **同步原语与它们的正确性保证**:
   - **锁**确保互斥访问——临界区不变式保证最多一个线程访问共享数据，RMW 序列被串行化
   - **屏障**确保阶段分离——强制所有线程在程序代码的特定点同步，使得数据更新按阶段对所有线程可见
   - **消息传递**通过通信而非共享来避免数据竞争，但要求仔细设计通信模式防止死锁
4. **常见陷阱**: 同步不足（分裂的原子操作导致重复处理）、过度同步（长时间持有锁导致饥饿）、死锁（循环等待——双向管道双方同时 `recv()` 是经典案例）。Coffman 的四个条件为死锁防护提供了理论基础
5. **异步任务**: `std::async`/`std::future` 提供高级的基于任务的并行模型。天真的递归并行化可能导致指数级任务爆炸（Fibonacci 的 4000x 开销是警示）。任务限制策略通过追踪活跃任务数在并行粒度和调度开销之间取得平衡
6. **Amdahl 定律**: 揭示了并行加速比的根本限制——由顺序部分的比例决定。即使完全可并行的部分被无限加速，总有 $$1/(1-p)$$ 的上限。快速排序在 128 个任务的限制下实现了 2.7x 加速比，远低于理论上的 6.67x 极限，说明缓存效应、伪共享和负载不均衡等现实因素的重要性
7. **Monte Carlo 方法**: 展示了"尴尬并行"问题的近乎理想的加速比。$$O(1/\sqrt{n})$$ 的收敛速度也提醒我们，概率算法的精度提升代价高昂

并发编程的核心洞察可以归结为一条铁律: **对共享可变状态的每一次访问，要么用锁将所有访问串行化，要么确保操作的原子性，要么完全放弃共享转而采用消息传递或不可变数据**。

三种策略各有适用场景:
- **共享内存 + 锁**: 适用于需要频繁、低延迟访问大型共享数据结构。代价是并发正确性的推理负担
- **消息传递**: 适用于天然按数据分区、通信模式规律的问题。代价是序列化/反序列化开销和潜在的通信延迟
- **不可变数据 + 函数式风格**: 最优雅的方案——通过消除可变性消除竞态条件的根源。代价是需要分配更多内存和可能更高的 GC 压力

无论选择哪种方案，对并发程序正确性的形式化推理——通过不变式、交错分析、归纳证明——都是确保程序在所有可能的执行路径下都正确的唯一可靠方法。
