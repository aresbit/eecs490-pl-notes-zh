# 第4章: 数据抽象 (Data Abstraction)

本章考察构造**抽象数据类型**（Abstract Data Type, ADT）的各类机制。ADT 允许我们将数据接口与数据实现分离开来，并借助**封装**（encapsulation）将数据与操作数据的过程绑定为一体。我们将从最基础的函数式数据建模出发，逐步演进到完整的面向对象编程、静态分析与动态类型、泛型机制，直至模块化组织。本章将对 vtable 调度、C3 线性化、菱形继承、CRTP 静态分派、Hindley-Milner 类型推断和明确赋值分析等核心推导进行完整的逐步展开，帮助读者从"是什么"深入到"为什么"。

## 4.1 函数式数据抽象 (Functional Data Abstraction)

面向对象并非构造数据抽象的唯一方式。我们首先利用函数作为基本构建块来建模数据，从一个简单的 pair（二元组）开始，逐步构造出列表、字典乃至基于消息传递的完整 ADT。本节的设计思路是：**用函数闭包来模拟对象，用函数参数来模拟方法调用**——理解这一点后，你会发现 OOP 在本质上只是函数式数据抽象的语法糖。

### 4.1.1 Pair 与列表

回顾 $\lambda$-演算中 pair 的 Church 编码：

$$\mathit{pair} = \lambda x.\ \lambda y.\ \lambda f.\ f\ x\ y$$

取第一个元素：$\mathit{first} = \lambda p.\ p\ \mathit{true}$，取第二个元素：$\mathit{second} = \lambda p.\ p\ \mathit{false}$。计算过程如下：将 `pair a b` 应用于 `first` 时，`first` 本身是 $\lambda p.\ p\ \mathit{true}$，所以 $(\mathit{pair}\ a\ b)\ \mathit{first} \to_\beta (\lambda f.\ f\ a\ b)\ \mathit{true} \to_\beta \mathit{true}\ a\ b \to_\beta a$。其中 $\mathit{true} = \lambda x.\ \lambda y.\ x$，因此 $\mathit{true}\ a\ b \to_\beta a$。

遵循同样的策略，我们可以在 Python 中以闭包实现不可变 pair：

```python
def pair(x, y):
    def get(i):
        return x if i == 0 else y
    return get

def first(p):
    return p(0)

def second(p):
    return p(1)
```

这里的 `pair()` 返回一个闭包函数 `get`，`x` 和 `y` 位于 `get` 的非局部作用域中。与 $\lambda$-演算用布尔值不同，Python 版本用整数索引（0 或 1）来提取元素。但这种实现不支持**变更**（mutation）。

为了实现可变 pair，我们使用 `nonlocal` 关键字来修改外层作用域的变量：

```python
def mutable_pair(x, y):
    def get(i):
        return x if i == 0 else y

    def set(i, value):
        nonlocal x, y
        if i == 0:
            x = value
        else:
            y = value

    return pair(get, set)  # 用不可变 pair 包装 get 和 set
```

此处使用不可变 `pair` 来返回 `get` 和 `set` 函数，避免了 `mutable_pair` 中的无限递归。可变 pair 的使用方式如下：

```python
>>> p = mutable_pair(3, 4)
>>> mutable_first(p)   # 等价于 first(p)(0)
3
>>> set_first(p, 5)    # 等价于 second(p)(0, 5)
>>> mutable_first(p)
5
```

**推导：闭包中的变量捕获**。`mutable_pair` 的核心机制是闭包：内层函数 `get` 和 `set` 捕获了外层 `mutable_pair` 的局部变量 `x` 和 `y`。当 `mutable_pair` 返回后，其栈帧本应被销毁，但由于闭包的存在，`x` 和 `y` 的生命周期被延长。`nonlocal` 声明至关重要——没有它，`set` 内部的 `x = value` 会创建一个新的局部变量，而非修改外层变量。Python 编译器的名字解析规则是：若名字在函数体内被赋值且未被 `global` 或 `nonlocal` 声明，则解析为局部变量。`nonlocal` 强制解析器回溯到外层作用域去定位名字。

### 4.1.2 消息传递 (Message Passing)

消息传递范式摒弃了在 ADT 外部定义操作函数的做法，转而将操作编码为**消息**，通过一个统一的 dispatch 函数分发：

```python
def mutable_pair(x, y):
    def dispatch(message, value=None):
        nonlocal x, y
        if message == 'first':
            return x
        elif message == 'second':
            return y
        elif message == 'set_first':
            x = value
        elif message == 'set_second':
            y = value
    return dispatch
```

此时 pair 完全表现为一个接受消息的函数：

```python
>>> p = mutable_pair(3, 4)
>>> p('first')          # 3
>>> p('set_first', 5)
>>> p('first')          # 5
```

消息传递的优点是**封装**：所有数据状态和操作都被封闭在 `dispatch` 闭包中，外部无法直接访问 `x` 和 `y`。这已经非常接近面向对象编程中向对象发送消息的语义模型——在 Smalltalk 的原始设计理念中，对象就是通过消息传递来完成所有交互的。

### 4.1.3 基于可变 pair 的列表

有了可变 pair，我们就能像 Scheme 一样，以 None 表示空表，用消息传递实现一个完整的可变列表：

```python
def mutable_list():
    empty_list = None
    head = empty_list
    tail = empty_list

    def size(mlist):
        if mlist is empty_list:
            return 0
        return 1 + size(mlist('second'))

    def getitem(mlist, i):
        if i == 0:
            return mlist('first')
        return getitem(mlist('second'), i - 1)

    def setitem(mlist, i, value):
        if i == 0:
            mlist('set_first', value)
        else:
            setitem(mlist('second'), i - 1, value)

    def append(value):
        nonlocal head, tail
        if head is empty_list:
            head = mutable_pair(value, empty_list)
            tail = head
        else:
            tail('set_second', mutable_pair(value, empty_list))
            tail = tail('second')

    def to_string():
        if head is empty_list:
            return '[]'
        return ('[' + str(head('first')) +
                to_string_helper(head('second')) + ']')

    def to_string_helper(mlist):
        if mlist is empty_list:
            return ''
        return (', ' + str(mlist('first')) +
                to_string_helper(mlist('second')))

    def dispatch(message, arg1=None, arg2=None):
        if message == 'len':
            return size(head)
        elif message == 'getitem':
            return getitem(head, arg1)
        elif message == 'setitem':
            return setitem(head, arg1, arg2)
        elif message == 'str':
            return to_string()
        elif message == 'append':
            return append(arg1)

    return dispatch
```

列表的底层表示是一个由可变 pair 级联而成的链式结构：

$$L = \mathit{pair}(a_0,\ \mathit{pair}(a_1,\ \dots\ \mathit{pair}(a_{n-1},\ \mathtt{None})\dots))$$

`append` 操作的复杂度为 $\mathcal{O}(1)$，因为我们维护了 `tail` 指针。但 `getitem` 和 `setitem` 操作为 $\mathcal{O}(i)$ 时间，因为它们需要沿链表遍历到第 $i$ 个元素。这是典型的用时间换空间的权衡——链表结构天然适合频繁的插入追加操作，而数组更适合随机访问。

注意我们用 `dispatch` 函数包装了所有内部操作。这代表一种设计模式：将「消息分发」从「业务逻辑」中分离，`size`、`getitem`、`setitem` 等函数处理逻辑，而 `dispatch` 只负责路由。这一思想在后文的 vtable 实现中也会反复出现。

### 4.1.4 字典

在列表之上，我们可以实现一个简单的字典 ADT，其内部是一个 key-value pair 构成的记录列表，使用线性搜索查找键：

```python
def dictionary():
    records = mutable_list()

    def get_record(key):
        size = records('len')
        i = 0
        while i < size:
            record = records('getitem', i)
            if key == record('first'):
                return record
            i += 1
        return None

    def getitem(key):
        record = get_record(key)
        return record('second') if record is not None else None

    def setitem(key, value):
        record = get_record(key)
        if record is None:
            records('append', mutable_pair(key, value))
        else:
            record('set_second', value)

    def dispatch(message, key=None, value=None):
        if message == 'getitem':
            return getitem(key)
        elif message == 'setitem':
            return setitem(key, value)

    return dispatch
```

此实现的时间复杂度为 $\mathcal{O}(n^2)$（每次操作都遍历整个记录列表）。若插入 $n$ 个键，每次 `setitem` 需检查已存在的键（平均搜索长度随 $n$ 增长），故总操作成本为 $\sum_{i=1}^{n} \mathcal{O}(i) = \mathcal{O}(n^2)$。引入迭代器抽象可将搜索从嵌套的递归调用改为线性的列表遍历，将每条操作的复杂度降为 $\mathcal{O}(n)$，但我们在这里保持最简单设计。

可对比 Python 内置字典使用的 `__getitem__` 和 `__setitem__` 特殊方法——Python 的内部实现使用哈希表实现 $\mathcal{O}(1)$ 平均查找时间，但我们的函数式 ADT 提供了几乎相同的接口（仅需 `d('getitem', key)` 而非 `d[key]`）。这说明：**接口与实现是完全分离的，同样的接口可以使用截然不同的底层数据结构**。

### 4.1.5 Dispatch 字典

条件分发在 ADT 复杂化后会变得臃肿。我们可以将消息到函数之间的映射存储在一个**dispatch 字典**中，由 dispatch 函数直接返回对应的内部函数，而后调用方自行传入参数：

```python
def account(initial_balance):
    def deposit(amount):
        new_balance = dispatch('getitem', 'balance') + amount
        dispatch('setitem', 'balance', new_balance)
        return new_balance

    def withdraw(amount):
        balance = dispatch('getitem', 'balance')
        if amount > balance:
            return 'Insufficient funds'
        balance -= amount
        dispatch('setitem', 'balance', balance)
        return balance

    def get_balance():
        return dispatch('getitem', 'balance')

    dispatch = dictionary()
    dispatch('setitem', 'balance', initial_balance)
    dispatch('setitem', 'deposit', deposit)
    dispatch('setitem', 'withdraw', withdraw)
    dispatch('setitem', 'get_balance', get_balance)

    def dispatch_message(message):
        return dispatch('getitem', message)

    return dispatch_message
```

使用方式展示了消息传递的间接调用模式：

```python
>>> a = account(33)
>>> a('get_balance')()     # 返回 33
>>> a('deposit')(4)        # 返回 37
>>> a('withdraw')(7)       # 返回 30
```

对比 Python 类实现的接口：`a.get_balance()`，`a.deposit(4)`。功能性 ADT 的接口与 OOP 类几乎一致，暗示两者在本质上共享相同的消息传递理念。这恰恰是面向对象编程的理论起源——在 Smalltalk 中，所有计算都通过对象间的消息传递完成，对象本身就是一组状态和响应消息的行为体的封装。

至此，我们完全用函数构建了一个 ADT 的层次体系：不可变 pair $\to$ 可变 pair $\to$ 列表 $\to$ 字典 $\to$ Dispatch 字典 $\to$ 银行账户。从下一节开始，我们将看到语言如何将这一范式内建成一级语言构造。

## 4.2 面向对象编程 (Object-Oriented Programming)

面向对象语言将 ADT 系统化地组织为**类（class）**，类定义了创建**对象（object）**的模板。OOP 提供四大支柱：

- **封装（Encapsulation）**：将数据与其操作函数捆绑在一起。类将字段和方法放在同一个命名单元中。
- **信息隐藏（Information hiding）**：限制对实现细节的访问。`private`/`protected` 等访问控制机制是具体手段。
- **继承（Inheritance）**：复用已有 ADT 的代码（实现继承）或接口（接口继承）。避免重复造轮子。
- **子类型多态（Subtype polymorphism）**：允许将派生类对象用于基类出现的位置，依赖**动态绑定（dynamic binding）**在运行时选择正确的函数实现。

### 4.2.1 成员 (Members)

对象由字段（fields / attributes / data members）和方法（methods / member functions）构成。在 C++ 中：

```cpp
class Foo {
public:
    int x;                    // 字段
    Foo(int x_);             // 构造函数
    int baz(int y);          // 成员函数
};
```

**静态字段** 在所有实例之间共享，属于类而非对象。以下是三种语言的对比，注意其语法和存储分配方式的差异：

```cpp
// C++: 需要类外定义（非编译期常量需要分配存储位置）
class Foo {
public:
    static int bar;
};
int Foo::bar = 3;  // 分配存储（必须在.cpp文件中，只能定义一次）
```

```java
// Java: 初始化在声明处或静态初始化块中完成
class Foo {
    static int bar = 3;
}
```

```python
# Python: 类体内直接定义即类属性；实例可通过同名属性遮蔽它
class Foo:
    bar = 3
```

静态成员可通过类名或实例访问（C++ 用 `::`，Java/Python 用 `.`），但通过实例访问容易让读者误以为是实例属性，因此通常推荐用类名访问。

### 4.2.2 访问控制 (Access Control)

信息隐藏要求限制成员的可见性。表 4.1 概括了多种语言中的访问级别：

| 访问范围 | public | private | C++ protected | Java protected | C# internal / Java 默认 | Python |
|------|--------|---------|---------------|----------------|--------------------------|--------|
| 同一实例 | Y | Y | Y | Y | Y | Y |
| 同一类 | Y | Y | Y | Y | Y | Y |
| 派生类 | Y | | Y | Y | | Y |
| 同一包/模块 | Y |   | | Y | Y | Y |
| 全局 | Y |   | | | | Y |

表中空白格表示该访问权限**不**授予该范围。可以看到 Python 的"全公开"哲学与其他语言形成鲜明对比——它依赖命名约定（`_` 前缀表示「内部使用」）而非编译器强制执行。

**protected 的微妙之处**：在 C++、C# 和 Java 中，派生类不能通过类型为基类的指针/引用访问 protected 成员，只能通过自身类型的指针/引用。原因在于：如果允许通过基类指针访问 protected 成员，派生类可以轻易编写一个方法，接受任意 `Base*` 参数并直接访问其 protected 数据，从而完全绕过了封装屏障。考虑以下 C++ 代码：

```cpp
class Base {
protected:
    int x = 4;
};

class Derived : public Base {
public:
    void foo(Base *b, Derived *d) {
        cout << b->x << endl;  // 编译错误: 不能通过 Base* 访问 protected 成员
        cout << d->x << endl;  // OK: d 是 Derived*，满足条件
    }
};
```

此处 `b->x` 被拒绝，因为编译器无法保证 `b` 指向的实际对象是 `Derived` 类型——即使它碰巧是，编译器也不会假设这一点。而 `d->x` 被允许，因为 `d` 的类型 `Derived*` 本身就是保护域的合法访问者。这一规则有时被称为"通过派生类型的访问规则"。

### 4.2.3 方法的种类

不同语言中，实例参数的传递方式有所不同：

- **C++**: `this` 是隐式参数，类型为 `T* const`（非 const 成员函数）或 `const T* const`（const 成员函数）。通常可省略，仅在名遮蔽时必须使用。
- **Python**: `self` 是显式参数，不可省略。这导致 Python 方法签名比 C++/Java 多一个参数，但也更明确。
- **静态方法**：不操作实例。在 C++ 用 `static` 关键字，Python 用 `@staticmethod` 装饰器。静态方法无法访问 `this`/`self`。
- **类方法**：接受类本身作为第一个参数（Python 中约定名为 `cls`），常用作工厂方法或实现各子类多态的构造函数。
- **属性方法**：伪装为字段访问的 getter/setter，Python 通过 `@property` 和 `@<name>.setter` 实现。

复数的极坐标/直角坐标转换是 property 的经典示例——内部存储直角坐标 (`real, imag`)，对外暴露 `magnitude` 和 `angle` 两个计算属性：

```python
import math

class Complex(object):
    def __init__(self, real, imag):
        self.real = real
        self.imag = imag

    @property
    def magnitude(self):
        return (self.real ** 2 + self.imag ** 2) ** 0.5

    @magnitude.setter
    def magnitude(self, mag):
        old_angle = self.angle     # 先获取当前角度
        self.real = mag * math.cos(old_angle)
        self.imag = mag * math.sin(old_angle)

    @property
    def angle(self):
        return math.atan2(self.imag, self.real)

    @angle.setter
    def angle(self, ang):
        old_magnitude = self.magnitude  # 先获取当前幅度
        self.real = old_magnitude * math.cos(ang)
        self.imag = old_magnitude * math.sin(ang)
```

这里 `magnitude` 和 `angle` 都是计算属性，设置其中任一都会更新底层存储的 `real` 和 `imag`。关键在于 **property 使得实现变更对客户代码完全透明**——如果将来改为存储 (`magnitude, angle`)，只需将 `real` 和 `imag` 变为 property，所有外部代码无需修改。这就是"面向接口编程，而非面向实现"的直接体现。

### 4.2.4 嵌套类与局部类

- **C++**: 嵌套类像普通类一样工作，但它们能访问外围类的 private 成员（反之不然，外围类需声明 `friend`）。局部类不能访问局部变量。C++ 的嵌套类机制本质上是名字访问权限的调整，不改变对象模型。
- **Java**: 非静态嵌套类（Inner class）隐式地持有一个对外围类实例的引用，因此可直接访问该实例的 `private` 字段。局部类可访问被声明为 "effectively final" 的局部变量（即虽未被 `final` 修饰但在初始化后不再被修改）。Java 编译器会将这种变量复制到内部类的构造函数中，因此要求它事实上不变。

```java
class Outer {
    private int x;
    Outer(int x) { this.x = x; }

    class Inner {
        private int y;
        Inner(int y) { this.y = y; }
        int get() { return x + y; }  // 直接访问 Outer.this.x，即使 x 是 private
    }
}
// 创建: Outer out = new Outer(3);
// Outer.Inner inn = out.new Inner(4);  注意 new 的语法: out.new
```

### 4.2.5 实现策略

OOP 的两种主要实现策略体现了静态效率与动态灵活性之间的经典权衡：

**(1) 基于记录的实现（Record-based）**：C++、Java（HotSpot）等效率优先的语言将对象的字段连续存储在内存中，每个字段在编译期解析为一个固定的偏移量。字段访问变为一次基址加偏移的计算，零运行时查找开销。以 64 位系统为例，假设 `sizeof(void*) = 8`，`sizeof(int) = 4`，`sizeof(double) = 8`：

```
偏移量 0:  vtable 指针 (8 bytes)
偏移量 8:  x (int, 4 bytes) + 4 bytes padding (对齐 y)
偏移量 16: y (double, 8 bytes)
总共:     24 bytes
```

编译器将 `f.x` 翻译为 `*(int*)((char*)&f + 8)`，将 `f.y` 翻译为 `*(double*)((char*)&f + 16)`。这是单条机器指令的开销。

**(2) 基于字典的实现（Dictionary-based）**：Python（CPython）将属性存储在 `__dict__` 哈希表中，支持运行时动态添加和删除属性。属性查找过程遵循确定的搜索链：先查实例的 `__dict__`，若未找到则查类的 `__dict__`，仍未找到则沿 MRO 向上搜索基类的 `__dict__`。这一灵活性以性能为代价——每次属性访问都需要哈希表查找。

Python 采取混合方案：默认使用字典，但可通过 `__slots__` 显式声明固定的属性集合，解释器随后为实例分配连续内存槽位：

```python
class Complex(object):
    __slots__ = ('real', 'imag')  # 实例不再有 __dict__，空间和访问都更高效
```

代价是失去运行时动态添加属性的能力（因为实例不再有字典）。此外，`__slots__` 的类不能用于多重继承中同时有 `__slots__` 的不同基类。

动态消息构造机制 `__getattribute__` 与 Java 的 Reflection API 相对应，二者都允许在运行时以字符串形式动态调用方法，是元编程的重要基础。

## 4.3 继承与多态 (Inheritance and Polymorphism)

### 4.3.1 继承的种类

**接口继承 vs 实现继承**：接口继承只复用接口约定（advertised protocol），实现继承复用内部实现。在大多数 OOP 语言中，实现继承总是伴随接口继承——派生类既可替换基类（子类型多态），也继承了基类的代码。C++ 是显著的例外，它通过 private 和 protected 继承来允许继承实现而不暴露接口：

**C++ 三种继承方式**：

| 继承方式 | 基类 `public` 成员在派生类中 | 基类 `protected` 成员 | 基类 `private` 成员 |
|----------|------------------------------|------------------------|---------------------|
| `public` | public | protected | 不可访问 |
| `protected` | protected | protected | 不可访问 |
| `private` | private | private | 不可访问 |

通用规则：继承后成员的可访问性 = $\min$(原始可访问性, 继承方式的可访问性)。这里的 `min` 按 `public > protected > private` 排序。因此 `private` 继承意味着"用基类的实现来构建我的实现，但从外部看，我不是基类的子类型"。

**抽象类与接口**：如果一个方法只有声明没有实现，则它是**抽象方法**（C++ 中称纯虚函数 `= 0`）。包含至少一个抽象方法的类是**抽象类**，不能直接实例化。Java 显式区分了 `abstract class`（可包含部分实现）和 `interface`（仅含抽象方法和常量，Java 8+ 可含 default 方法）：

```java
abstract class A {
    abstract void foo();
}

interface I {
    void bar();  // 隐式为 public abstract
    default void baz() { System.out.println("default"); } // Java 8+
}

class C extends A implements I {
    void foo() { System.out.println("C.foo()"); }
    public void bar() { System.out.println("C.bar()"); }
}
```

**Mixin** 是另一种继承模式：模块方法被混入到类中而不建立子类型关系。Ruby 的 `include` 是典型实现。这避免了菱形继承问题，体现了"组合优于继承"的部分理念。

```ruby
class Counter
  include Comparable
  attr_accessor :count
  def initialize()
    @count = 0
  end
  def increment()
    @count += 1
  end
  def <=>(other)
    @count <=> other.count
  end
end
# 实现 <=> 后，自动获得 <, <=, ==, >, >=, between? 等方法
```

### 4.3.2 类层次结构

- **Java / Python**：所有类隐式派生自根类（`Object` / `object`），形成单根的树结构（Java 单继承）或有向无环图（Python 允许多重继承）。
- **C++**：无根类。`void*` 可指向任意对象，但不提供任何行为（连析构都不能调用）。

单根结构的一个实际好处是可以编写适用于所有对象的容器和算法（如 `Vector<Object>`），而 C++ 要实现同等功能则需要模板（`vector<T*>`）或 `void*` + 类型擦除技巧。

### 4.3.3 方法重写与动态绑定

**重写（override）**是子类型多态的关键机制。要点是：
- C++ 必须用 `virtual` 关键字显式声明方法支持动态绑定（否则默认静态绑定）。
- Java 所有实例方法默认使用动态绑定（`final`、`private` 和 `static` 方法除外）。
- Python 和 Ruby 等动态类型语言必然支持动态绑定，因为没有静态类型信息可用。

**协变返回类型（Covariant return type）**：重写方法可以返回原返回类型的子类型。C++ 和 Java 都支持。例如 Java 的 `Object.clone()` 返回 `Object`，但派生类可声明返回自身类型。

**逆变参数（Contravariant parameter）问题**：语义上，重写方法接受比基类方法更宽泛的参数类型（逆变）是安全的。但在支持重载的语言中，逆变参数的方法会被视为**重载（overload）**（Java）或**隐藏（hide）**（C++），这会导致不可预料的行为。为防范意外，Java 提供 `@Override` 注解，C++11 提供 `override` 关键字——编译器在编译期即可检测非重写的错误。

### 4.3.4 动态绑定的实现：vtable

在基于记录的语言中，动态绑定通过**虚函数表（virtual table, vtable）**实现。理解 vtable 是理解 OOP 性能模型的关键。考虑以下 C++ 类层次：

```cpp
struct A {
    int x;
    double y;
    virtual void a();         // vtable[0]
    virtual int b(int i);    // vtable[1]
    virtual void c(double d); // vtable[2]
    void f();                 // 非虚函数 — 不会进入 vtable
};

struct B : A {
    int z;
    char w;
    virtual void d();         // vtable[3] (新增)
    virtual double e();       // vtable[4] (新增)
    virtual int b(int i);    // vtable[1] (重写 A::b — 注意偏移不变!)
    void f();                 // 隐藏 A::f — 非虚，不进入 vtable
};
```

下面是 64 位系统上的精确内存布局（假设 `sizeof(void*) = 8`, `sizeof(int) = 4`, `sizeof(double) = 8`）：

**A 的对象布局：**

```
    A object (size = 24):
   ┌──────────────┐  offset 0
   │ vtable ptr   │ ────→  A's vtable
   ├──────────────┤  offset 8
   │ x (int)      │
   │ padding (4B) │  (对齐到 8 字节边界)
   ├──────────────┤  offset 16
   │ y (double)   │
   └──────────────┘

    A's vtable:
   ┌──────────────┐
   │ &A::a  [0]   │  (A 的第一个 virtual 方法按声明顺序)
   ├──────────────┤
   │ &A::b  [1]   │
   ├──────────────┤
   │ &A::c  [2]   │
   └──────────────┘
```

重要细节：非虚函数 `A::f` **不在 vtable 中**。编译器对 `a.f()` 直接生成 `call A::f`，无需经过 vtable。

**B 的对象布局：**

```
    B object (size = 32):
   ┌──────────────┐  offset 0
   │ vtable ptr   │ ────→  B's vtable
   ├──────────────┤  offset 8
   │ x (int)      │  (继承自 A，偏移量 8 — 与 A 完全相同)
   │ padding (4B) │
   ├──────────────┤  offset 16
   │ y (double)   │  (继承自 A，偏移量 16 — 与 A 完全相同)
   ├──────────────┤  offset 24
   │ z (int)      │  (B 新增)
   │ w (char)     │  (B 新增，1 字节)
   │ padding (3B) │  (对齐到 8 字节)
   └──────────────┘

    B's vtable:
   ┌──────────────┐
   │ &A::a  [0]   │  (从 A 继承，未重写)
   ├──────────────┤
   │ &B::b  [1]   │  (重写 A::b — 与 A::b 在 vtable 中占用相同偏移)
   ├──────────────┤
   │ &A::c  [2]   │  (从 A 继承，未重写)
   ├──────────────┤
   │ &B::d  [3]   │  (B 新增)
   ├──────────────┤
   │ &B::e  [4]   │  (B 新增)
   └──────────────┘
```

**关键不变式**：基类字段在派生类对象中具有与基类对象中**完全相同的偏移量**。这使得 `A*` 和 `B*` 指针可以安全地访问 `x` 和 `y`，而无需任何指针调整。同样，基类的 vtable 布局在派生类 vtable 的前缀部分**完全保留**——新方法追加在现有布局之后。

#### vtable 调度步骤推导：完整过程

以调用 `aptr->b()` 为例，其中 `A *aptr = new B;`：

**第 1 步——编译期的静态类型查找**：编译器看到 `aptr->b()`，首先检查 `aptr` 的**静态类型**（即变量声明的类型）`A`。在 `A` 的类定义中查找名为 `b` 的成员，并执行重载解析（若有多个同名函数）。这里只有一个 `int b(int)`，确定为被调用的方法。

**第 2 步——检查 virtual 属性**：编译器检查方法 `b` 在类 `A` 中的声明——它被标记为 `virtual`。因此编译器不能生成直接调用指令，而需要通过 vtable 进行间接调用。

**第 3 步——确定 vtable 偏移量**：编译器查阅 `A` 的 vtable 布局，确定 `A::b` 位于 `vtable[1]`（索引从 0 开始，对应于 `sizeof(void*) * 1 = 8` 字节的偏移）。

**第 4 步——生成间接调用代码**：编译器生成的代码（以伪 C 表示）如下：

```c
// 步骤 a: 从对象首部加载 vtable 指针
void **vtable_ptr = *(void ***)aptr;

// 步骤 b: 按编译期确定的偏移量从 vtable 中加载函数指针
void *func_ptr = vtable_ptr[1];  // 偏移量在编译期固定为 1

// 步骤 c: 通过函数指针调用，传入隐含的 this (即 aptr 本身)
((int (*)(A *, int))func_ptr)(aptr, 实际参数);
```

**第 5 步——运行时实际执行**：`aptr` 在运行时指向一个 `B` 类型的对象。该对象的首 8 字节存储着**指向 `B` 的 vtable 的指针**。因此：
- `vtable_ptr` = B 的 vtable 的地址
- `vtable_ptr[1]` = `&B::b`（因为 `B::b` 重写了 `A::b`，位于 B 的 vtable 的同一个槽位 `[1]`）
- 实际调用的是 `B::b`，传入的 `this` 是 `aptr` 的值（指向 B 对象首部）

**为什么 this 指针正确**：无论通过 `A*` 还是 `B*` 访问 `B` 对象，对象首部地址相同。`B` 的内存布局保证了继承自 `A` 的部分在对象的最前部，因此 `A*` 和 `B*` 在数值上相等。`B::b` 的函数体可以安全地通过 `this` 访问任何 A 和 B 的字段，因为所有字段的偏移量在编译期已经以 B 的布局为准确定。

**性能分析**：整个 dispatch 过程需要 **两次内存间接访问**（load vtable ptr, load func ptr）加一次间接函数调用。而非虚函数调用只需要编译期确定的直接跳转指令（一条 `call` 或 `jmp`）。这就是为什么 C++ 默认使用非虚方法——它为不需要多态的场景提供了零开销的调用机制。

### 4.3.5 多重继承

#### 基于字典的实现：C3 线性化

Python 使用 **C3 线性化**算法计算类的方法解析顺序（Method Resolution Order, MRO）。C3 保证了单调性和局部优先序——一个类的 MRO 在其所有父类的 MRO 中保持相同的相对顺序。

以 `Butterfly(Insect, WingedAnimal)` 为例：

```python
class Animal:
    def defend(self):
        print('run away!')

class Insect(Animal):
    pass

class WingedAnimal(Animal):
    def defend(self):
        print('fly away!')

class Butterfly(Insect, WingedAnimal):
    pass
```

**C3 算法的形式化定义**：对于类 $C$ 及其父类列表 $B_1, B_2, \ldots, B_n$，令 $L(C)$ 表示 $C$ 的线性化（MRO），则：

$$L(C(B_1, B_2, \ldots, B_n)) = [C] \oplus \mathit{merge}(L(B_1), L(B_2), \ldots, L(B_n), [B_1, B_2, \ldots, B_n])$$

其中 $\mathit{merge}$ 操作的定义如下：取所有输入列表中第一个列表的首部 $h$，检查 $h$ 是否**不出现在任何后续列表的尾部**（即不出现在除头部以外的任何位置）。若通过检查，则将 $h$ 移出并追加到结果中；若失败，则尝试下一个列表的首部。重复此过程直至所有列表为空。若某一轮没有任何头部可通过检查，则引发类型错误（表明继承层次不一致）。

**推导 $L(\mathit{Insect})$**：

已知 $L(\mathit{Animal}) = [\mathit{Animal}, \mathit{object}]$，$L(\mathit{object}) = [\mathit{object}]$（Python 中所有类的 MRO 尾部都以 `object` 结束）。

$$\begin{aligned}
L(\mathit{Insect}) &= [\mathit{Insect}] \oplus \mathit{merge}(L(\mathit{Animal}), [\mathit{Animal}]) \\
&= [\mathit{Insect}] \oplus \mathit{merge}([\mathit{Animal}, \mathit{object}], [\mathit{Animal}])
\end{aligned}$$

合并过程：

- **轮次 1**：取首列表首部 $\mathit{Animal}$，检查尾部。$\mathit{Animal}$ 出现在第二个列表 `[Animal]` 中 —— 但它在第二个列表的**头部**，不是尾部（尾部列表为空 `[]`）。因此通过检查。移除 $\mathit{Animal}$。结果：`[Butterfly→正在推导, 先看Insect]` 不对，我们重新来。

严格来说：
$$\mathit{merge}([\mathit{Animal}, \mathit{object}], [\mathit{Animal}])$$

- 取第一个列表的首部 = $\mathit{Animal}$。
- 检查 $\mathit{Animal}$ 是否出现在其余列表中非头部的位置。
  - 在第二个列表 `[Animal]` 中，$\mathit{Animal}$ 位于头部，不在尾部。
  - **通过**。
- 移出 $\mathit{Animal}$，追加到结果。剩余列表：$([\mathit{object}], [])$。
- 取 $\mathit{object}$ 作为首部，无后续列表。**通过**。

因此 $L(\mathit{Insect}) = [\mathit{Insect}, \mathit{Animal}, \mathit{object}]$。

同理 $L(\mathit{WingedAnimal}) = [\mathit{WingedAnimal}, \mathit{Animal}, \mathit{object}]$。

**推导 $L(\mathit{Butterfly})$ 的核心合并过程**：

$$\begin{aligned}
L(\mathit{Butterfly}) = [\mathit{Butterfly}] \oplus \mathit{merge}(
&[\mathit{Insect}, \mathit{Animal}, \mathit{object}], \\
&[\mathit{WingedAnimal}, \mathit{Animal}, \mathit{object}], \\
&[\mathit{Insect}, \mathit{WingedAnimal}]
)
\end{aligned}$$

现在执行 merge：

| 轮次 | 候选首部 | 列表中 | 检查是否在其余列表尾部 | 结果 | 剩余列表 |
|------|---------|--------|----------------------|------|---------|
| 1 | Insect (来自 L1) | L1[0] | L2 尾部=[Animal,object] 不含 Insect; L3 尾部=[WingedAnimal] 不含 Insect | 通过 | 追加 Insect | L1=[Animal,object]; L2=[WingedAnimal,Animal,object]; L3=[WingedAnimal] |
| 2 | Animal (来自 L1) | L1[0] | L2=[WingedAnimal,**Animal**,object]: Animal 在尾部! | **失败** | — | — |
| 2 | WingedAnimal (来自 L2) | L2[0] | L1 尾部=[object] 不含 WingedAnimal; L3 尾部=[] 不含 WingedAnimal | 通过 | 追加 WingedAnimal | L1=[Animal,object]; L2=[Animal,object]; L3=[] |
| 3 | Animal (来自 L1) | L1[0] | L2=[Animal,object]: Animal 在头部，不在尾部(**尾部=[object]**). 通过 | 通过 | 追加 Animal | L1=[object]; L2=[object] |
| 4 | object (来自 L1) | L1[0] | L2=[object]: object 在头部而非尾部. 通过 | 通过 | 追加 object | L1=[]; L2=[] |

最终结果：

$$MRO(\mathit{Butterfly}) = [\mathit{Butterfly}, \mathit{Insect}, \mathit{WingedAnimal}, \mathit{Animal}, \mathit{object}]$$

**语义解释**：C3 保证了 `WingedAnimal` 排在 `Animal` 之前——这是因为 `WingedAnimal` 是更特化的类（它重写了 `defend`）。在第 2 轮时，`Animal` 无法通过检查，因为 `WingedAnimal` 作为更特化的类必须排在它前面。最终 `Butterfly().defend()` 调用 `WingedAnimal.defend()`，打印 `'fly away!'`。如果使用朴素的深度优先搜索，结果将是 `Animal.defend`（打印 `'run away!'`），这显然不是程序员期望的行为。

#### 基于记录的多重继承：视图与 this 指针调整

在 C++ 的记录式实现中，多重继承打破了「所有字段相对于对象首部的偏移一致」的前提。考虑：

```cpp
struct A { int x;   virtual void a(); virtual void b(); };
struct B { int y;   virtual void c(); virtual void d(); };
struct C : A, B { int z;   virtual void a(); virtual void c(); virtual void e(); };
```

C 的对象需要同时支持 `A*`、`B*` 和 `C*` 三种视角。C++ 的解决方案是为每个非主基类维护一个独立的**视图**，每个视图有自己的 vtable 指针：

```
  完整的 C 对象内存布局 (64-bit):

  offset  内容                           指向的视角
  ──────  ─────────────────────────────  ────────────────
  0       vtable ptr (view A, C)  ────→ C's vtable (view A, C)
  8       x (int, 来自 A)
  12      padding (4B)
  16      vtable ptr (view B)     ────→ C's vtable (view B)
  24      y (int, 来自 B)
  28      padding (4B)
  32      z (int, 来自 C)
  36      padding (4B)  (对齐到 8B 边界)
  ──────
  Total: 40 bytes (含对齐)
```

**this 指针调整的精确推导**：当编译器将 `C*` 赋值给 `B*` 或作为函数参数传递时：

```cpp
C *c_ptr = new C();       // c_ptr = 0x1000 (假设，指向对象首部)
A *a_ptr = c_ptr;          // a_ptr = 0x1000 (A 是主基类，无需调整)
B *b_ptr = c_ptr;          // 编译器插入:
                           // b_ptr = (B*)((char*)c_ptr + 16)
                           //       = 0x1010
```

偏移量 16 字节 = `sizeof(vtable_ptr) + sizeof(int) + 4 bytes padding` = 8 + 4 + 4 = 16。

这就是 B 视图距离对象首部的偏移。现在，若通过 `B*` 指针调用方法，所有 B 视图内的字段访问都基于 `b_ptr` 计算偏移量——`b_ptr->y` 翻译为偏移 8（B 的 vtable ptr 之后），相对于 `b_ptr` 的地址 0x1010，即实际地址 0x1018 = 对象中的第 24 字节，正好是 `y` 的位置。完全正确。

但真正的挑战出现在**通过 B 视图调用 C 重写的方法**时。考虑：

```cpp
void C::c() {
    cout << z;  // z 在 C 中偏移 32 (对象首部)
}
```

如果通过 `b_ptr` 调用 `c()`，`this` 指针的值是 0x1010（B 视图的地址）。但 `C::c()` 中用偏移 32 来访问 `z`——基于 0x1010 的偏移 32 是 0x1030，这是错误的位置！

**解决方案：vtable 条目中的 this 调整**。vtable 条目不只存储函数指针，还存储一个**偏移量**，指示调用前需要对 `this` 做的调整：

```
  C's vtable (view A, C):           C's vtable (view B):
  ┌──────────────────┬──────┐       ┌──────────────────┬──────┐
  │ &C::a            │   0  │       │ &C::c            │ -16  │  ← 需要将 B 视点调整 -16 字节回到 C 视点
  ├──────────────────┼──────┤       ├──────────────────┼──────┤
  │ &A::b            │   0  │       │ &B::d            │   0  │
  ├──────────────────┼──────┤       └──────────────────┴──────┘
  │ &C::c            │   0  │
  ├──────────────────┼──────┤
  │ &B::d            │ +16  │  ← A→B 视点: this += 16
  ├──────────────────┼──────┤
  │ &C::e            │   0  │
  └──────────────────┴──────┘
```

调用 `b_ptr->c()` 的实际运行时逻辑：

```c
// b_ptr = 0x1010
vtable_entry = b_ptr->vtable[0];          // 获取 {func_ptr, this_adjust}
this_adjusted = (char*)b_ptr + vtable_entry.this_adjust;  // 0x1010 + (-16) = 0x1000
vtable_entry.func_ptr(this_adjusted);     // 调用 C::c, this = 0x1000 ✓
```

**Thunk 优化**：在实际编译器中（如 GCC、Clang、MSVC），vtable 条目通常不显式存储偏移量，而是存储一个指向 **thunk** 的指针——thunk 是一段编译器生成的微型代码：

```asm
; thunk for C::c via B view
    sub  rdi, 16        ; 调整 this 指针 (在 System V AMD64 ABI 中 this 通过 rdi 传递)
    jmp  C::c           ; 跳转到真正的方法
```

vtable 条目直接指向这个 thunk。这种设计避免了在每个调用点复制指针调整代码，且与现代处理器的分支预测和指令缓存配合良好。

#### 菱形问题与虚继承

经典菱形继承问题：一个类通过两条路径继承同一个祖先类。C++ 提供两种处理方式——**重复继承**（默认）和**虚继承**。

**重复继承的内存布局**：

```cpp
class Animal { std::string name; };  // sizeof(string) ≈ 32 (取决于实现)
class Insect : public Animal { int i; };
class WingedAnimal : public Animal { int w; };
class Butterfly : public Insect, public WingedAnimal { };
```

```
  Butterfly 对象 (重复继承, 含两个独立的 Animal 子对象):

  offset 0:    Insect's vtable ptr       (8B)
  offset 8:    Animal::name (来自 Insect) (32B)  ← 第一份 Animal
  offset 40:   Insect::i                  (4B + padding → 8B)
  offset 48:   WingedAnimal's vtable ptr  (8B)
  offset 56:   Animal::name (来自 Winged) (32B)  ← 第二份 Animal (不同!)
  offset 88:   WingedAnimal::w            (4B + padding → 8B)
  总计: 96 bytes
```

此布局下 `Butterfly` 有两个 `name` 字段——这通常是浪费且语义错误的（一只蝴蝶只有一个名字）。

**虚继承的内存布局**：通过 `virtual` 关键字声明共享基类：

```cpp
class Insect : public virtual Animal { int i; };
class WingedAnimal : public virtual Animal { int w; };
class Butterfly : public Insect, public WingedAnimal { };
```

虚继承的对象布局引入了额外的间接层。一种常见实现（Itanium C++ ABI / GCC / Clang）如下：

```
  Butterfly 对象 (虚继承):

  offset 0:    Insect's vtable ptr       ──→ Insect-in-Butterfly vtable
  offset 8:    Insect::i                  (4B + 4B padding)
  offset 16:   WingedAnimal's vtable ptr ──→ WingedAnimal-in-Butterfly vtable
  offset 24:   WingedAnimal::w            (4B + 4B padding)
  offset 32:   Animal::name              (32B)  ← 唯一的 Animal 子对象
  offset 64:   vbase offset (Animal)     (记录 Animal 相对 Insect/WingedAnimal 的偏移)

  vtable 中包含 virtual base offset，运行时访问 name 的路径：
  Insect* → vptr → vtable[i] → virtual base offset → this + offset → name
```

虚继承的代价是：每次通过 `Insect*` 或 `WingedAnimal*` 访问 `Animal` 的成员时，都需要额外的一次 vtable 查表来获取虚基类的偏移量。这是"共享"在实现上的成本。更重要的是，虚继承通常要求中间类（`Insect`、`WingedAnimal`）在设计之初就预见到菱形问题的发生并使用 `virtual` 继承——这在一定程度上打破了基类与派生类之间的抽象隔离。

## 4.4 静态分析 (Static Analysis)

编译器在不运行程序的情况下对源代码进行的分析称为**静态分析**。由于 Rice 定理表明图灵完备语言的程序行为分析（任意非平凡语义性质）是不可判定的，静态分析只能给出近似结果——它必然在某些情况下出错。设计静态分析的核心抉择在于：

### 4.4.1 可靠性 vs 完备性

设在某种行为特性下，程序集合 $P$ 中的正确程序为 $T \subseteq P$，分析器判定为"正确"的集合为 $A \subseteq P$。

- **可靠的分析（sound）**：$A \subseteq T$。即分析器说"正确"的程序必定正确（没有假阳性），但可能错误地拒绝正确的程序（假阴性 false negative）。
- **完备的分析（complete）**：$T \subseteq A$。即正确的程序必被接受（没有假阴性），但可能错误地接受不正确的程序（假阳性 false positive）。

Rice 定理意味着不可能存在既可靠又完备的分析。实际系统的设计选择是：
- 编译期静态分析通常倾向于**可靠性**——宁可拒绝合法程序（要求程序员修改代码以满足分析器的保守要求），也不放过潜在错误。
- 运行时动态分析（如数组越界检查）通常倾向于**完备性**——允许所有合法执行，在有违例时通过运行时异常来处理。

在现实中，由于语言复杂度极高（如 C++ 的模板、宏、未定义行为等），许多实用分析器（如 Clang Static Analyzer）实际上**既不严格可靠也不严格完备**——它们在两个方向上都做出了实用性妥协。

### 4.4.2 类型等价

类型等价回答"这两个类型是否相同"的问题，直接影响类型检查的正确性：

- **结构等价（structural equivalence）**：类型由结构定义，结构相同的类型等价。例如两个分别定义的 `record { int a; int b; }` 被视为相同类型。ML 系语言使用结构等价。
- **名等价（name equivalence）**：每个类型声明产生一个全新类型。即使两个类有完全相同的字段，它们也互不兼容。Java、C++（类类型）使用名等价。
- **严格 vs 宽松名等价**：严格名等价认为别名也是不同类型（`typedef double weight;` 后 `weight` 和 `double` 不等价）；宽松名等价（C/C++ 实际采用）认为别名与原始类型相同。严格名等价有助于捕捉语义错误（如混淆 weight 和 height），但增加了程序员负担。

大多数现代语言在基本类型上使用名等价（或宽松名等价），在泛型/模板实例化上采用某种形式的结构等价。

### 4.4.3 类型兼容性

严格类型等价过于严格——现实中我们经常需要将一种类型用于另一种类型的预期上下文中。语言通过**类型兼容性**规则放宽这一限制：

- **子类型多态**：若 $S <: T$，则 $S$ 可安全地用于 $T$ 的上下文。Liskov 替换原则是子类型的语义基础。
- **类型强制转换（type coercion）**：编译器隐式插入类型转换。典型例子是整数到浮点数的提升（`int` $\to$ `double`）。
- **const 限定规则（C++）**：
  - 非 const 左值可强制为 const 左值（添加 const 是安全的——缩小了修改权限）。
  - const 左值不可强制为非 const 左值（需要显式 `const_cast`，因为这可能引起未定义行为）。
  - const 左值可强制为非 const 右值（复制不改变原对象，安全）。

### 4.4.4 类型推断

类型推断是编译器确定每个表达式类型的机制。在 ML 系语言中，**Hindley-Milner (HM) 类型推断**是最优雅的解决方案之一。让我们完整推导一个复杂的多态高阶函数的类型。

**示例：推导 $f = \lambda x.\ \lambda y.\ (+\ x\ y)$ 的类型**

1. **初始类型变量赋值**：
   - $x: \alpha$，$y: \beta$，$f$ 的类型暂时记为 $\gamma$
   
2. **分析外层的 lambda**：
   - $f = \lambda x.\ body$，所以 $\gamma = \alpha \to \delta$，其中 $\delta$ 为 body 的类型
   - body $= \lambda y.\ (+\ x\ y)$，所以 $\delta = \beta \to \epsilon$，其中 $\epsilon$ 为 $(+\ x\ y)$ 的类型

3. **分析内层表达式** $(+\ x\ y)$：
   - $+$ 的类型：在 Standard ML 中为 $\mathit{int} \to \mathit{int} \to \mathit{int}$（注意 SML 的 `+` 只工作在 `int` 上）
   - $(+\ x)$ 的类型：应用后 $x$ 必须为 $\mathit{int}$，所以 $\alpha = \mathit{int}$，$(+\ x)$ 的类型为 $\mathit{int} \to \mathit{int}$
   - $(+\ x\ y)$ 的类型：应用后 $y$ 必须为 $\mathit{int}$，所以 $\beta = \mathit{int}$，整个表达式类型为 $\mathit{int}$

4. **统一所有约束**：
   - $\alpha = \mathit{int}$，$\beta = \mathit{int}$，$\epsilon = \mathit{int}$
   - $\gamma = \alpha \to \beta \to \epsilon = \mathit{int} \to \mathit{int} \to \mathit{int}$

5. **最终推断结果**：$f: \mathit{int} \to \mathit{int} \to \mathit{int}$

**HM 算法的核心机制**是**合一（unification）**：将类型变量的约束系统通过等式的传递闭包求解，得到最一般的类型（most general type）。任何具体类型都是最一般类型的特化实例。

在 OCaml 中，由于 `+` 被定义为多态（通过 `+` 关联的模块类型约束），`let f x y = x + y` 会被推断为 `int -> int -> int`（因为 `+` 默认作用于整数），但 `let g x y = x +. y` 会被推断为 `float -> float -> float`。

**C++ 中的类型推导**：
- `auto` 遵循模板类型推导规则：丢弃引用性，丢弃顶层 const/volatile（但保留底层 const）。
- `decltype(expr)` 返回表达式的精确类型，包括引用性和 const 限定。
- C++14 起，函数返回类型可以完全省略，由 `auto` 从返回语句推断。

**Java 的类型推断**：Java 8+ 的 lambda 表达式使用**目标类型推断**——参数类型由赋值目标（函数式接口）确定。编译器从周围的上下文推导 lambda 的期望类型，再回头检查 lambda 体是否满足该类型。

### 4.4.5 控制流分析

#### 明确赋值分析 (Definite Assignment)

Java 语言规范要求：每个局部变量在使用前**必须**已被明确赋值。编译器使用保守的数据流分析来证明这一点。让我们展开一个复杂案例的完整推理：

**案例 1：编译器拒绝的代码**

```java
class Foo {
    public static void main(String[] args) {
        int i;
        if (args.length > 0) {     // 分支点 1
            i = args.length;
        }
        if (args.length <= 0) {    // 分支点 2
            i = 0;
        }
        System.out.println(i);     // 使用点
    }
}
```

**编译器的分析轨迹**（按控制流图的路径）：

```
[程序入口]
  i: 未赋值 (definitely unassigned)

→ [第一个 if 语句]
  ├─ true 分支 (args.length > 0):
  │    i = args.length  → i 变为 "已赋值"
  │    → 合并点 A: i 的状态 = "可能已赋值" (取两条路径的保守上界)
  └─ false 分支 (args.length <= 0):
       i 保持未赋值
       → 合并点 A: i 的状态 = "可能已赋值"

→ [第二个 if 语句] (从合并点 A 出发, i 的状态为 "可能已赋值")
  ├─ true 分支 (args.length <= 0):
  │    i = 0  → i 变为 "已赋值"
  │    → 合并点 B: i 的状态 = "可能已赋值" (保守上界)
  └─ false 分支 (args.length > 0):
       i 保持 "可能已赋值"
       → 合并点 B: i 的状态 = "可能已赋值"

→ [println(i)]: 此时 i 状态 = "可能已赋值" ≠ "明确已赋值"
  → 编译错误: variable i might not have been initialized
```

关键点：编译器**不跟踪条件之间的逻辑关系**。它看到两个 `if` 各自都可能不执行其分支体，因此在合并点采用保守策略。虽然语义上 $p \lor \neg p$ 永真（两个条件中必有一真），但编译器不做命题逻辑推理。

**案例 2：编译器接受的修正**

```java
int i;
if (args.length > 0) {
    i = args.length;
} else {
    i = 0;
}
System.out.println(i);
```

此时分析结果：
- true 分支：`i` 被赋值。
- false 分支：`i` 被赋值。
- 合并后（if-else 保证恰好一个分支执行）：`i` = **明确已赋值**。
- 到达 `println(i)`：通过检查。

**案例 3：含循环的复杂情形**

```java
int i;
while (condition) {
    i = getValue();
}
System.out.println(i);  // 错误: i 可能未赋值
```

循环体的执行不是保证的——若 `condition` 初始为假，循环体一次也不执行。因此编译器拒绝此代码。正确的做法是在声明时初始化：`int i = 0;`。

**案例 4：分析器可证明的循环**

```java
int i;
while (true) {
    i = getValue();
    if (someCondition) break;
}
// 编译器仍拒绝: 虽然 break 可能不执行，但 while(true) 入口到 println 的路径
// 经过的是循环条件（恒真），而 i 在循环体内赋值——但 break 之后呢？
```

实际上这段代码的明确赋值分析相当微妙。标准的 Java 编译器接受以下形式：

```java
int baz(int x) {
    while (true) {
        if (x < 0) {
            return 0;  // 保证从这里退出函数
        }
    }
}
```

编译器能证明 `while(true)` 唯一退出路径是通过 `return`，因此函数总能正常返回。

#### 可达性分析

Java 中不可达语句是编译错误（而非警告）。编译器通过类似的控制流分析检测这类问题——这些规则与明确赋值分析对称：保证每条路径都到达 return 的分析，本质上就是寻找未到达 return 的不安全路径的过程。

## 4.5 动态类型与 RTTI (Dynamic Typing and RTTI)

除了动态绑定之外，语言和运行时系统还以其他形式使用**运行时类型信息**（Run-Time Type Information, RTTI），并允许程序员通过特定机制访问它。RTTI 实现了静态类型系统无法处理的一类操作——即在运行时根据对象的实际类型做决策。

### 4.5.1 C++ RTTI

**`dynamic_cast`**：在运行时安全地将基类指针/引用转换为派生类指针/引用。其实现依赖 vtable——因此仅适用于包含至少一个虚函数的类（多态类型）。

```cpp
struct A { virtual void bar() {} };
struct B : A {};

void foo(A *a) {
    if (B *b = dynamic_cast<B*>(a)) {
        // a 指向的实际动态类型是 B（或其派生类）
        b->B_specific_method();
    } else {
        cout << "not a B" << endl;
    }
}
// dynamic_cast<B*>: 成功 → 非空指针; 失败 → nullptr
// dynamic_cast<B&>: 成功 → 正常; 失败 → 抛出 std::bad_cast
```

内部实现原理：`dynamic_cast` 通过遍历 vtable 中存储的类型信息（通常是 `type_info` 对象或类层次结构描述符），沿着继承图进行指针调整和类型比较。

**`typeid` 运算符**：返回 `std::type_info` 对象。如果操作数是多态类型（至少有一个虚函数），则返回**动态类型**的信息；否则返回静态类型。

```cpp
#include <typeinfo>
const type_info &ti = typeid(*a);  // 若 A 是多态的，返回实际动态类型
cout << ti.name() << endl;         // 实现相关（如 GCC: "1B" 表示 B）
```

### 4.5.2 Java RTTI

- `instanceof` 运算符：`obj instanceof String` 在运行时检查对象类型
- `getClass()` 方法（定义在 `Object` 上）：返回 `Class` 对象，支持完整的反射操作
- 所有对象类型转换在运行时都会被动态检查（不仅是 `instanceof`），失败时抛出 `ClassCastException`

### 4.5.3 Python RTTI

- `isinstance(obj, type)` 函数：检查继承关系
- `type(obj)` 返回对象的精确类型（不检查继承关系，`type(obj) == SomeClass` 不考虑子类）
- Python 中类型是一等实体，因此 RTTI 天然集成在语言中

### 4.5.4 Java 数组协变问题

Java 在引入泛型之前，为了支持对任意数组的多态操作，制定了数组的协变规则：若 $A <: B$ 则 $A[] <: B[]$。这造成了一个著名的类型安全漏洞：

```java
String[] sarray = {"foo", "bar"};
Object[] oarray = sarray;        // 合法: String[] <: Object[] (协变)
oarray[0] = "Hello";             // 合法: String 存入 Object[] — 运行时类型一致
oarray[1] = new Integer(3);      // 编译通过: Integer <: Object
                                 // 运行时抛出 ArrayStoreException!
// 如果没有运行时检查，下一步:
sarray[1].length();              // 将调用 Integer.length() — 方法不存在!
```

Java 的补救措施是：在**每次向数组元素写入时**由虚拟机动态检查元素的实际类型与数组的动态类型是否兼容。这是空间换时间的典型案例——每次写操作都需要额外的类型检查开销。更好的设计本应是使用参数多态（泛型），但 Java 1.0 并不支持泛型，而这一设计缺陷因向后兼容性要求被永久固化在语言规范中。

## 4.6 泛型 (Generics)

**参数多态**（parametric polymorphism）允许一段代码在不同类型上工作，而不依赖子类型关系和动态绑定。与子类型多态的「运行时动态选择」不同，参数多态通常在编译期完成类型的确定和代码生成。

### 4.6.1 隐式参数多态

ML 系语言（OCaml、Haskell、Standard ML）通过 HM 类型推断对未标注类型的函数自动推导多态类型：

```ocaml
let max x y = if x > y then x else y;;
(* 推断类型: 'a -> 'a -> 'a  (在 OCaml 中, 因为 > 是多态的) *)
```

编译器生成**一份**代码，使用通用的机器表示（如统一的字长）处理所有类型。在 OCaml 中，多态函数的参数通过统一表示（boxed values）传递，值的具体类型在运行时不参与计算。

### 4.6.2 显式参数多态

C++ 的 `template` 是实现显式参数多态的经典机制。与 ML 的隐式多态不同，C++ 模板使用**代码生成**策略：

```cpp
template <typename T>
T max(const T &x, const T &y) {
    return x > y ? x : y;
}
```

**非类型参数**：模板参数不仅可以是类型，还可以是编译期常量值：

```cpp
template <typename T, int N>
class array {
    T data[N];  // N 必须是编译期常量
};
// 使用: array<double, 5> arr;
```

**约束的设计哲学差异**：
- **隐式约束（C++）**：检查推迟到模板实例化时刻。错误消息包含了模板展开后的全部细节，极难解读。优点是灵活，缺点是用户体验差。
- **显式约束（Java/C#）**：泛型定义时通过 `extends` 声明边界。定义处即可检查泛型体代码是否合规；实例化时仅需验证实参满足边界。错误消息精确。
- **C++20 Concepts**：为 C++ 引入了显式约束的能力，弥补了隐式约束的历史缺陷。

### 4.6.3 实现策略

参数多态的实现分裂为两大阵营，各有适用场景：

| 策略 | 代表语言 | 实现方式 | 优点 | 缺点 |
|------|---------|---------|------|------|
| **类型擦除** | Java, ML | 生成一份代码，类型参数替换为 Object/通用表示 | 二进制体积小，互操作简单 | 不能使用基本类型（Java），运行时开销（装箱拆箱） |
| **代码生成** | C++ | 每个类型实参组合生成独立代码 | 零运行时开销，可与基本类型无缝工作 | 代码膨胀，编译时间增长，需要源码可见 |

Java 类型擦除的后果不仅限于性能——它导致泛型在运行时不可见（`List<String>.class` 不存在），且不允许通过 `new T()` 创建类型参数实例，也不允许创建泛型数组（`new T[10]`）。这些都是语言设计者对向后兼容性的妥协。

### 4.6.4 Java 泛型

默认情况下 Java 泛型将类型参数替换为 `Object`，因此泛型体内部只能调用 `Object` 的方法。使用 `extends` 进行约束：

```java
static <T extends Comparable<T>> T max(T x, T y) {
    return x.compareTo(y) > 0 ? x : y;
}
```

**PECS 模式（Producer Extends, Consumer Super）**是理解 Java 型变（variance）的关键记忆法：
- `? extends T`：生产者——可以从结构中**读取** `T` 或其子类。
- `? super T`：消费者——可以向结构中**写入** `T` 或其父类。

具体到泛型约束的经典场景：

```java
// 问题: Square extends Rectangle implements Comparable<Rectangle>
// 而非 Comparable<Square>
// 用上界 T extends Comparable<T> 会导致 Foo<Square> 编译失败

// 修正: 使用 ? super T
class Foo<T extends Comparable<? super T>>
    implements Comparable<Foo<T>> { ... }
```

此时 `Square` 满足边界：`Square` 实现 `Comparable<Rectangle>`，而 `Rectangle` 是 `Square` 的超类（`? super Square` 的合法候选），故通过检查。这是 Java 型变系统中最精妙也最令人困惑的部分。

### 4.6.5 CRTP：奇异递归模板模式

CRTP（Curiously Recurring Template Pattern）允许 C++ 在编译期实现类似 mixin 的代码复用，且**完全消除虚函数调用开销**。这是静态多态（也称模拟动态绑定）的标准实现技术：

```cpp
template<class T>
class Comparable {
public:
    bool operator<(const T &rhs) const {
        // static_cast 在编译期完成 — 零运行时开销
        return static_cast<const T*>(this)->compare(rhs) < 0;
    }
    bool operator<=(const T &rhs) const {
        return static_cast<const T*>(this)->compare(rhs) <= 0;
    }
};

class Counter : public Comparable<Counter> {
    int count = 0;
public:
    void increment() { ++count; }
    int get_count() const { return count; }
    // 注意: compare 不是虚函数!
    int compare(const Counter &other) const {
        return count - other.count;
    }
};
```

使用方式与普通的比较操作无异：`Counter c1, c2; if (c1 < c2) {...}`。

**CRTP vs vtable dispatch — 性能推导（概念层面的汇编对比）**：

传统 vtable dispatch（假设 `compare` 声明为 `virtual`）的执行路径：

```asm
; c1 < c2 (vtable 方式)
mov  rax, [c1]           ; 1. 加载 vptr (内存读 #1)
mov  rax, [rax + 24]     ; 2. 加载 vtable[3] — compare 的函数指针 (内存读 #2)
                          ;    (假设 compare 在 vtable 偏移 24)
mov  rdi, c1_addr        ; 3. 设置 this 指针
mov  rsi, c2_addr        ; 4. 设置 rhs 参数
call rax                 ; 5. 间接调用 (分支预测困难)
```

CRTP static dispatch 的执行路径：

```asm
; c1 < c2 (CRTP 方式)
mov  rdi, c1_addr        ; 1. 设置 this
mov  rsi, c2_addr        ; 2. 设置 rhs
call Counter::compare    ; 3. 直接调用 (编译期确定地址)
cmp  eax, 0              ; 4. 比较返回值与 0
setl al                  ; 5. 设置 operator< 的结果
```

CRTP 节省了 vptr 的存储空间（每个对象减少 8 字节），消除了两次内存间接访问，且直接函数调用可被内联展开——编译器甚至可能将 `count - other.count < 0` 整体内联到调用处的计算中，实现零函数调用开销。

**适用条件**：CRTP 的前提是 `static_cast<const T*>(this)` 安全执行——即 `this` 实际上指向的对象确实是 `T` 类型。在 CRTP 模式中，这一前提由模板实例化自然保证：`Comparable<Counter>` 仅当被 `Counter` 继承时才有意义。CRTP 在 Microsoft ATL/WTL 中被大量用于构建轻量级 COM 组件和 Windows UI 框架。

### 4.6.6 鸭子类型

在动态类型语言中，函数自动适用于任何支持所需操作的类型——这就是**鸭子类型**（duck typing）。其核心是：不检查类型声明，只检查对象是否拥有被调用的方法。

```python
def max(x, y):
    return x if x > y else y  # 任何定义了 __gt__ 方法的类型都可以
```

鸭子类型的力量在于极致的灵活——你不需要通过继承或接口声明来承诺行为。然而代价也是明显的：方法名相同不保证语义相同——`run()` 在 `Athlete` 中可能表示"起跑"，在 `Thread` 中表示"启动线程"，而在 `Script` 中表示"执行"。在静态类型语言中可以轻易通过类型签名区分的行为，在鸭子类型中只能依靠文档约定和调用者的常识，错误要到运行时才会暴露。

## 4.7 模块与命名空间 (Modules and Namespaces)

抽象数据类型聚焦于**单个类型**的封装，而实际系统总是由许多相互依赖的类型、函数和常量构成——这就是**模块**要解决的问题。模块化将系统分解为高内聚、低耦合的单元，使得大型程序的维护成为可能。

### 4.7.1 翻译单元

**翻译单元（translation unit）**是独立编译的基本单位，通常对应一个源文件。在 C/C++ 中还包括所有递归 `#include` 的文件内容。翻译单元是独立编译的边界——每个翻译单元被编译为一个目标文件，随后由链接器将这些目标文件合并为最终的可执行程序。

C++ 的头文件/源文件分离模式体现了声明与定义的经典分离：

```cpp
// Triangle.hpp — 接口 (声明)
class Triangle {
    double a, b, c;
public:
    Triangle();
    Triangle(double, double, double);
    double area() const;
    double perimeter() const;
    void scale(double s);
};

// Triangle.cpp — 实现 (定义)
#include "Triangle.hpp"
double Triangle::area() const { return a * b * c; }
// ... 其他成员函数的定义
```

Java 和 Python 没有头文件的概念：编译器或解释器从源文件（或字节码文件）中自动提取声明信息。C++ 中需要头文件的根本原因是 C++ 编译模型要求：要使用一个类型，编译器必须知道它的大小和布局。前向声明只在使用指针/引用时足够。

### 4.7.2 Python 模块与包

Python 的模块即一个 `.py` 文件，构成独立的作用域：

```python
from math import sqrt     # 导入单个名字到当前作用域
import sys                # 导入模块对象本身
sys.argv                  # 通过属性访问模块内的名字
```

**包** 是模块的目录级组织，通过 `__init__.py` 文件标识目录为包（Python 3.3+ 也支持无 `__init__.py` 的命名空间包）。包的典型结构如下：

```
sound/                    # 顶级包
    __init__.py           # 包的初始化代码
    formats/              # 子包 (文件格式转换)
        __init__.py
        wavread.py
        wavwrite.py
    effects/              # 子包 (音效处理)
        __init__.py
        echo.py
        surround.py
    filters/              # 子包 (滤波器)
        __init__.py
        equalizer.py
```

导入一个包时会执行其 `__init__.py`，这可以用于包的初始化或控制 `from package import *` 的行为（通过 `__all__`）。

### 4.7.3 Java 包

Java 的组织体系与 Python 类似但语法不同。`package` 声明位于文件首行，`import` 声明紧随其后。Java 支持单名导入和通配符导入，且支持静态导入（将类的静态方法和常量导入当前作用域）。

### 4.7.4 C++ 命名空间

C++ 使用 `namespace` 来组织名字空间，多个命名空间块可以定义同属一个命名空间的实体：

```cpp
namespace foo {
    struct A {};
    int x;
}
namespace foo {  // 扩展 foo 命名空间
    struct B : A {};
}
// 使用: foo::A *a = new foo::A;
using foo::A;          // 导入单个名字
using namespace foo;   // 导入整个命名空间 (应谨慎, 易引发命名冲突)
```

全局命名空间中的名字可通过 `::name` 引用。Java 中未声明包名的类进入匿名包（anonymous/default package），类似于 C++ 的全局命名空间。

### 4.7.5 链接与信息隐藏

C 语言提供了最低层次的封装机制——通过 `static` 关键字限制名字的可见性：

- **内部链接（static）**：名字仅在当前翻译单元可见。用于函数和全局变量的信息隐藏。
- **外部链接（默认）**：名字跨翻译单元可见，但要求整个程序中有且仅有一个定义（One Definition Rule）。变量声明和定义的精微之处在于：`int count;` 也是定义（包含默认初始化），而 `extern int count;` 是纯声明。

**C 的不透明类型模式** 是实现 ADT 的经典 C 惯用法：

```c
// stack.h — 只提供前向声明和操作接口
typedef struct list *stack;
stack stack_make();
void stack_push(stack s, int i);
int stack_top(stack s);
void stack_pop(stack s);

// stack.c — 完整定义仅在此可见
struct node {
    int datum;
    struct node *next;
};
struct list {
    struct node *first;
};
```

外部代码只能通过指针操作 `stack`，无法直接访问其内部表示。这是**编译期封装**——语言本身没有 `private` 关键字，但通过限制定义的可见性达到了等价的信息隐藏效果。

### 4.7.6 模块初始化

跨模块/翻译单元的初始化顺序是常见的陷阱：

- **Python 的循环导入**可能导致 `AttributeError`。Python 执行模块顶层代码在首次 `import` 时，若模块 A 导入 B 且 B 又导入 A，B 在 A 的代码执行完之前尝试访问 A 的名字，则得到 `AttributeError`。解决方案包括延迟导入（在函数体内 `import`）或将共同依赖提取到第三个模块。
- **Java** 的静态初始化在首次使用类时触发，顺序由运行时动态决定，不应依赖跨类的初始化顺序。
- **C++** 的静态存储期变量初始化：同一翻译单元内按定义顺序；跨翻译单元的顺序**未指定**，且在动态链接库场景下情况更复杂。这可能导致"静态初始化顺序灾难"（static initialization order fiasco）——一个翻译单元的全局对象在构造时使用了另一个尚未构造的翻译单元的全局对象。C++ 的 `constinit`（C++20）和函数内静态变量（首次执行到时初始化）是常见的缓解手段。
